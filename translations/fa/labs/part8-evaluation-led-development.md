# بخش ۸: توسعه مبتنی بر ارزیابی با Foundry Local

> **هدف:** ساخت یک چارچوب ارزیابی که به صورت سیستماتیک عوامل هوش مصنوعی شما را آزمون و امتیازدهی می‌کند، با استفاده از همان مدل محلی هم به عنوان عامل تحت آزمایش و هم به عنوان داور، تا بتوانید با اطمینان روی پرامپت‌ها تکرار کنید قبل از ارسال.

## چرا توسعه مبتنی بر ارزیابی؟

هنگام ساخت عوامل هوش مصنوعی، «ظاهراً درست است» کافی نیست. **توسعه مبتنی بر ارزیابی** خروجی‌های عامل را مانند کد در نظر می‌گیرد: ابتدا تست‌ها را می‌نویسید، کیفیت را اندازه می‌گیرید و فقط وقتی امتیازها به آستانه رسیدند، منتشر می‌کنید.

در Zava Creative Writer (بخش ۷)، **عامل ویرایشگر** قبلاً به عنوان یک ارزیاب سبک عمل می‌کند؛ تصمیم می‌گیرد قبول یا بازبینی شود. این آزمایشگاه آن الگو را به یک چارچوب ارزیابی تکرارپذیر رسمی تبدیل می‌کند که می‌توانید به هر عامل یا خط لوله‌ای اعمال کنید.

| مشکل | راه‌حل |
|---------|----------|
| تغییرات پرامپت به صورت نامحسوس کیفیت را خراب می‌کند | **داده مجموعه طلایی** افت کیفیت را شناسایی می‌کند |
| تعصب «روی یک نمونه کار می‌کند» | **موارد تست متعدد** موارد لبه‌ای را نشان می‌دهد |
| ارزیابی کیفیت ذهنی | **امتیازدهی مبتنی بر قانون + مدل زبانی بزرگ به عنوان داور** اعداد ارائه می‌دهد |
| راهی برای مقایسه نسخه‌های پرامپت نیست | **اجرای ارزیابی کنار هم** با امتیازهای تجمیع شده |

---

## مفاهیم کلیدی

### ۱. داده‌های مجموعه طلایی

یک **داده مجموعه طلایی** مجموعه‌ای گردآوری شده از موارد تست با خروجی‌های شناخته شده و مورد انتظار است. هر مورد تست شامل:

- **ورودی**: پرامپت یا سوالی که برای عامل ارسال می‌شود
- **خروجی مورد انتظار**: آنچه پاسخ صحیح یا با کیفیت بالا باید شامل شود (کلمات کلیدی، ساختار، حقایق)
- **دسته‌بندی**: گروه‌بندی برای گزارش‌گیری (مثلاً «دقت واقعی»، «لحن»، «کامل بودن»)

### ۲. بررسی‌های مبتنی بر قانون

بررسی‌های سریع و قطعی که نیاز به مدل زبانی بزرگ ندارند:

| بررسی | موضوع آزمون |
|-------|--------------|
| **محدوده طول** | پاسخ نه خیلی کوتاه (تنبل) و نه خیلی بلند (حاشیه‌ای) باشد |
| **کلمات کلیدی مورد نیاز** | پاسخ شامل عبارات یا مفاهیم مورد انتظار باشد |
| **اعتبارسنجی فرمت** | JSON قابل پارس شدن باشد، سرفصل‌های Markdown حضور داشته باشند |
| **محتوای ممنوعه** | نام‌های تجاری توهمی نباشد، اشاره به رقبا نشده باشد |

### ۳. مدل زبانی بزرگ به عنوان داور

استفاده از **همان مدل محلی** برای امتیازدهی به خروجی‌های خود (یا خروجی‌های یک نسخه دیگر از پرامپت). داور موارد زیر را دریافت می‌کند:

- سوال اصلی
- پاسخ عامل
- معیارهای امتیازدهی

و یک امتیاز ساختاریافته برمی‌گرداند. این الگو شبیه الگوی ویرایشگر در بخش ۷ است اما به صورت سیستماتیک در سراسر مجموعه تست اعمال می‌شود.

### تمرین ۲ - درک داده مجموعه طلایی

موارد تست تعریف‌شده در نمونه ارزیابی را بررسی کنید. هر مورد تست شامل:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**سوالات برای بررسی:**
۱. چرا مقادیر مورد انتظار **کلمات کلیدی** هستند و جملات کامل نیستند؟
۲. چند مورد تست برای یک ارزیابی قابل اعتماد لازم دارید؟
۳. چه دسته‌بندی‌هایی را برای کاربرد خود اضافه می‌کنید؟

---

### تمرین ۳ - درک امتیازدهی مبتنی بر قانون در مقابل مدل زبانی بزرگ به عنوان داور

چارچوب ارزیابی از **دو لایه امتیازدهی** استفاده می‌کند:

#### بررسی‌های مبتنی بر قانون (سریع، قطعی)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### مدل زبانی بزرگ به عنوان داور (ظریف، کیفی)

همان مدل محلی با یک رویکرد امتیازدهی ساختاریافته به عنوان داور عمل می‌کند:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**سوالات برای بررسی:**
۱. چه زمانی به بررسی‌های مبتنی بر قانون نسبت به داوری مدل زبانی اعتماد می‌کنید؟
۲. آیا یک مدل می‌تواند به طور قابل اعتماد خروجی خود را قضاوت کند؟ محدودیت‌ها چیست؟
۳. این با الگوی عامل ویرایشگر از بخش ۷ چه تفاوتی دارد؟

---

### تمرین ۴ - مقایسه نسخه‌های پرامپت

نمونه، **دو نسخه پرامپت** را روی همان موارد تست اجرا می‌کند:

| نسخه | سبک پرامپت سامانه |
|---------|-------------------|
| **پایه** | کلی: «شما یک دستیار مفید هستید» |
| **تخصصی شده** | دقیق: «شما یک کارشناس Zava DIY هستید که محصولات خاصی را توصیه و راهنمایی مرحله به مرحله ارائه می‌دهید» |

پس از اجرا، کارت امتیازی مانند زیر مشاهده خواهید کرد:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**تمرین‌ها:**
۱. ارزیابی را اجرا کنید و امتیازهای هر نسخه را یادداشت کنید
۲. پرامپت تخصصی شده را حتی خاص‌تر کنید. آیا امتیاز بهتر می‌شود؟
۳. نسخه سوم پرامپت اضافه کنید و هر سه را مقایسه کنید.
۴. نام مدل را تغییر دهید (مثلاً `phi-4-mini` در مقابل `phi-3.5-mini`) و نتایج را مقایسه کنید.

---

### تمرین ۵ - اعمال ارزیابی روی عامل خودتان

چارچوب ارزیابی را به عنوان قالبی برای عوامل خود به کار ببرید:

۱. **داده مجموعه طلایی خود را تعریف کنید**: ۵ تا ۱۰ مورد تست با کلمات کلیدی مورد انتظار بنویسید.
۲. **پرامپت سامانه خود را بنویسید**: دستورالعمل‌هایی که برای عامل می‌خواهید تست کنید.
۳. **ارزیابی را اجرا کنید**: امتیازهای پایه را بگیرید.
۴. **تکرار کنید**: پرامپت را تنظیم کنید، دوباره اجرا و مقایسه کنید.
۵. **آستانه تعیین کنید**: مثلاً «امتیاز ترکیبی زیر ۰.۷۵ منتشر نشود».

---

### تمرین ۶ - ارتباط با الگوی ویرایشگر Zava

به عامل ویرایشگر Zava Creative Writer بازگردید (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# ویرایشگر یک مدل زبان بزرگ به عنوان داور در تولید است:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

این همان **مفهوم** مدل زبانی بزرگ به عنوان داور در بخش ۸ است، اما در خط تولید به جای مجموعه تست آفلاین تعبیه شده است. هر دو الگو:

- خروجی JSON ساختاریافته از مدل می‌گیرند.
- معیارهای کیفیت تعریف‌شده در پرامپت سامانه را اعمال می‌کنند.
- تصمیم قبول/رد می‌گیرند.

**تفاوت:** ویرایشگر در تولید اجرا می‌شود (روی هر درخواست). چارچوب ارزیابی در توسعه اجرا می‌شود (قبل از انتشار).

## نکات کلیدی

| مفهوم | برداشت |
|---------|----------|
| **داده‌های مجموعه طلایی** | موارد تست را زود شناخته و جمع‌آوری کنید؛ این‌ها تست‌های رگرسیون هوش مصنوعی شما هستند |
| **بررسی‌های مبتنی بر قانون** | سریع، قطعی و خطاهای واضح را می‌گیرند (طول، کلمات کلیدی، فرمت) |
| **مدل زبانی بزرگ به عنوان داور** | امتیازدهی دقیق کیفیت با استفاده از همان مدل محلی |
| **مقایسه پرامپت** | چندین نسخه را روی همان مجموعه تست اجرا کنید تا بهترین را انتخاب کنید |
| **مزیت اجرای محلی** | همه ارزیابی‌ها به صورت محلی اجرا می‌شود: بدون هزینه API، بدون محدودیت نرخ، بدون خروج داده از دستگاه |
| **ارزیابی قبل از انتشار** | آستانه کیفیت تعیین کنید و انتشار را بر اساس امتیاز ارزیابی کنترل کنید |

## گام‌های بعدی

- **افزایش مقیاس**: موارد تست و دسته‌بندی‌ها را به داده مجموعه طلایی‌تان اضافه کنید.
- **خودکارسازی**: ارزیابی را در خط لوله CI/CD خود بگنجانید.
- **داورهای پیشرفته**: از مدل بزرگ‌تر به عنوان داور در حالی که خروجی مدل کوچک‌تر را تست می‌کنید استفاده کنید.
- **ردیابی در طول زمان**: نتایج ارزیابی را ذخیره کنید تا در بین نسخه‌های پرامپت و مدل مقایسه کنید.
