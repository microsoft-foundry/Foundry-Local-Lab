# အပိုင်း 8: Foundry Local ဖြင့် အကဲဖြတ်မှု ခေါင်းဆောင်သော ဖွံ့ဖြိုးတိုးတက်မှု

> **ရည်ရွယ်ချက်:** AI ကိုယ်စားလှယ်များကို စနစ်တကျ စမ်းသပ်ခြင်းနှင့် အမှတ်ပေးခြင်းအတွက် အကဲဖြတ်မှု ဇယားတစ်ခု တည်ဆောက်ပါ၊ စမ်းသပ်နေသော ကိုယ်စားလှယ်နှင့် အကဲဖြတ်သူပင် တူညီသော တည်နေရာမော်ဒယ်ကို အသုံးပြု၍ သင် prompt များကို ယုံကြည်စိတ်ချစွာ သွားလာနိုင်စေရန်။

## အကဲဖြတ်မှု ခေါင်းဆောင် ဖွံ့ဖြိုးတိုးတက်မှု အဘယ်ကြောင့်လဲ?

AI ကိုယ်စားလှယ်များ ဖန်တီးရာတွင် "တကယ်မှန်တယ်လို့ မြင်ရတယ်" ဆိုတာ မလုံလောက်ပါ။ **အကဲဖြတ်မှု ခေါင်းဆောင် ဖွံ့ဖြိုးတိုးတက်မှု** သည် ကိုယ်စားလှယ်ထွက်ကုန်များကို ကုဒ်လိုမျိုး ကြည့်ခြင်းဖြစ်သည်။ သင် ပထမဦးစွာ စမ်းသပ်ချက်များရေးပြီး အရည်အသွေးတိုင်းတာ၊ အမှတ်များ သတ်မှတ်ချက်ကို ဖြတ်လျှင်သာ ပို့ဆောင်ပါသည်။

Zava Creative Writer (အပိုင်း 7) တွင် **တည်းဖြတ်သူ ကိုယ်စားလှယ်** သည် အနည်းငယ် အကဲဖြတ်သူအဖြစ် လုပ်ဆောင်ပြီး ယူလူ့သော ACCEPT သို့မဟုတ် REVISE ဆုံးဖြတ်သည်။ ဤဓါတ်မှန်ကို လက်တွေ့အသုံးပြုနိုင်သော အကဲဖြတ်မှု ဇယားအဖြစ် တိတိကျကျ ထုတ်လုပ်ထားသည်။

| ပြဿနာ | ဖြေရှင်းချက် |
|---------|----------|
| Prompt ပြောင်းလဲမှုကြောင့် အရည်အသွေး ချွတ်ယွင်းခြင်း | **ရွှေ dataset** သည် ပြန်လည်ကျဆင်းမှုများကို တွေ့ရှိသည် |
| "တစ်ဥပမာဖြင့် အလုပ်လုပ်သည်" ကြောင့် ထုံးစံများ (bias) ဖြစ်ခြင်း | **စမ်းသပ်ထားသော အကြိမ်များ** သည် ထူးခြားမှုများကို ဖော်ထုတ်သည် |
| ပုဂ္ဂိုလ်ရေး အရည်အသွေး ထောက်ခံချက် | **စည်းမျဉ်းအခြေခံ + LLM-အဖြစ် အကဲဖြတ်သူ scoring** သည် နံပါတ်များ ပေးသည် |
| Prompt များကို နှိုင်းယှဉ်ဖို့ နည်းမရှိခြင်း | **ဘက်နှစ်ဖက် Eval လည်ပတ်မှုများ** နှင့် စုစုပေါင်းအမှတ်များ ရရှိရန် |

## အဓိက အယူအဆများ

### 1. ရွှေ Dataset များ

**ရွှေ dataset** ဆိုသည်မှာ ရည်ရွယ်ချက် သိသာပြီဖြစ်သော စမ်းသပ်ချက်အစုအဝေးတစ်ခုဖြစ်သည်။ တစ်ခုချင်းစမ်းသပ်ချက်တွင် ပါဝင်သည့်အချက်များမှာ -

- **Input**: ကိုယ်စားလှယ်ထံ ပို့လိုသော prompt သို့မဟုတ် မေးခွန်း
- **မျှော်မှန်း output**: မှန်ကန်သော နှင့် အရည်အသွေးမြင့် အဖြေတစ်ခုတွင် ယူဆရသော အချက်များ (keywords, ဖွဲ့စည်းမှု, သတင်းအချက်အလက်)
- **အမျိုးအစား**: အစီရင်ခံမှုအတွက် အုပ်စုခြင်း (ဥပမာ "သိပ္ပံမှန်ကန်မှု", "အသံအထား", "ပြည့်စုံမှု")

### 2. စည်းမျဉ်းအခြေခံ စစ်ဆေးမှုများ

LLM မလိုအပ်ပဲ မြန်ဆန်ပြီး သေချာသော စစ်ဆေးချက်များ -

| စစ်ဆေးချက် | စမ်းသပ်သည့်အရာ |
|-------|--------------|
| **အရှည်အကန့်အသတ်များ** | အဖြေသည် အလွန်တို (အလယ်ကျသော) သို့မဟုတ် အလွန် ရှည်လျား (စကားလုံးတိုက်) မဟုတ်ရ။ |
| **လိုအပ်သည့် keywords** | အဖြေတွင် မျှော်မှန်းသော စကားလုံးများ သို့မဟုတ် တစ်ခုခု ပါဝင်သည် |
| **ဖော်မတ် စစ်ဆေးမှု** | JSON ဟာ ဖတ်နိုင်ပြီး၊ Markdown header များရှိသည် |
| **တားမြစ်ထားသော အကြောင်းအရာ** | မသေချာသော trade name များ မပါ၊ ပြိုင်ဘက်အကြောင်းမပါဝင်ရ |

### 3. LLM-အဖြစ် အကဲဖြတ်သူ

**တူညီသော တည်နေရာမော်ဒယ်** ကို ကိုယ်ထွက်များ၏ အရည်အသွေးကို အမှတ်ပေးရန် အသုံးပြုပါ (သို့မဟုတ် prompt ဝါရင့်ကွဲ မတူသော output များ)။ အကဲဖြတ်သူ လက်ခံသည် -

- မူရင်းမေးခွန်း
- ကိုယ်စားလှယ်၏ အဖြေ
- အမှတ်ပေးသတ်မှတ်ချက်များ

ပြီးသောအခါ ဖွဲ့စည်းထားသော အမှတ်တစ်ခု ပြန်ပေးတယ်။ ဤအချက်မှာ အပိုင်း 7 မှ တည်းဖြတ်သူ ပုံစံနှင့် ဆန့်ကျင်ဘက်ဖြစ်သော်လည်း စနစ်တကျ စမ်းသပ်မှု စုစည်းမှုတစ်ခုအတွင်း အသုံးပြုသည်။

### 4. အကဲဖြတ်အားဖြင့် ဆက်လက်တိုးတက်ရေး လည်ပတ်မှု

### လေ့ကျင့်ခန်း 2 - ရွှေ Dataset ကို နားလည်ရန်

အကဲဖြတ်မှု နမူနာရှိ စမ်းသပ်ချက်များကို စစ်ဆေးကြည့်ပါ။ တစ်ခုချင်းစမ်းသပ်ချက်တွင် ပါဝင်သည် -

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**စဉ်းစားရန် မေးခွန်းများ:**
1. မျှော်မှန်းသော ဥပမာများသည် ပြည့်စုံသော စာကြောင်းများ မဟုတ်ဘဲ **keywords** ကျရော?
2. ယုံကြည်စိတ်ချစွာ အကဲဖြတ်ရန် စမ်းသပ်ချက် များကို ဘယ်လောက်အများလိုအပ်သလဲ?
3. ကိုယ့် application အတွက် ဘယ်အမျိုးအစားများ ထည့်သွင်းလိုသလဲ?

### လေ့ကျင့်ခန်း 3 - စည်းမျဉ်းအခြေခံ နောက် LLM-အဖြစ် အကဲဖြတ်သူ အမှတ်ပေးခြင်း ကို နားလည်ရန်

အကဲဖြတ်မှု ဇယားတွင် **အမှတ်ပေး အဆင့်နှစ်ခု** အသုံးပြုသည် -

#### စည်းမျဉ်းအခြေခံ စစ်ဆေးမှုများ (မြန်ဆန်ပြီး သေချာသော)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-အဖြစ် အကဲဖြတ်သူ (အသေးစိတ်နှင့် အရည်အသွေးအခြေခံ)

တူညီသော တည်နေရာမော်ဒယ်သည် ဖွဲ့စည်းထားသော rubric ဖြင့် အကဲဖြတ်သူအဖြစ် ပါဝင်သည် -

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**စဉ်းစားရန် မေးခွန်းများ:**
1. ဘယ်အချိန်မှာ LLM-အဖြစ် အကဲဖြတ်သူထက် စည်းမျဉ်းအခြေခံ စစ်ဆေးမှု 믿ယူမလဲ?
2. မော်ဒယ်တစ်ခုက ကိုယ့် output ကို ယုံကြည်စိတ်ချစွာ အကဲဖြတ်နိုင်သလား? ကန့်သတ်ချက်များကဘာများလဲ?
3. အပိုင်း 7 မှ ညွှန်ပြသူ ကိုယ်စားလှယ် ပုံစံနှင့် ဘယ်လို နှိုင်းယှဥ်ပြီး ပြင်ဆင်ထားသလဲ?

### လေ့ကျင့်ခန်း 4 - Prompt များနှိုင်းယှဉ်ခြင်း

နမူနာတွင် **prompt နှစ်မျိုး** ကို တူညီသော စမ်းသပ်ချက်နှင့် လည်ပတ်သည် -

| အမျိုးအစား | စနစ် prompt ပုံစံ |
|---------|-------------------|
| **အခြေခံ** | ပုံမှန်: "သင်သည် ကူညီမှု ကူညီသူတစ်ဦးဖြစ်သည်" |
| **အထူးပြု** | အသေးစိတ်: "သင်သည် Zava DIY အထူးပြု လက်တွေ့ပညာရှင်ဖြစ်သည်၊ ထုတ်ကုန်များအကြောင်း အကြံပြုပါ သော လိုက်နာဆောင်ရွက်နည်းများကို သီးခြားညွှန်ပြသည်" |

လည်ပတ်ပြီးနောက် အမှတ်စာရင်း ဒီလို ဖော်ပြမှာ ဖြစ်ပါတယ် -

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**လေ့ကျင့်ခန်းများ:**
1. အကဲဖြတ်မှု လည်ပတ်ပြီး အမျိုးအစားစုံအတွက် အမှတ်များ မှတ်သားပါ။
2. အထူးပြု prompt ကို ပို တိကျအောင် ပြင်ဆင်ပါ။ အမှတ်တိုးလာမလား?
3. နောက် prompt တစ်ခု ထပ်ထည့်ပြီး အားလုံးကို နှိုင်းယှဉ်ပါ။
4. မော်ဒယ် alias (ဥပမာ `phi-4-mini` နှင့် `phi-3.5-mini`) ကိုပြောင်း၍ ရလဒ်များ နှိုင်းယှဉ်ပါ။

### လေ့ကျင့်ခန်း 5 - ကိုယ်ပိုင် ကိုယ်စားလှယ် သို့ အကဲဖြတ်မှု ကိုအသုံးချခြင်း

ကိုယ်ပိုင် ကိုယ်စားလှယ်များအတွက် အကဲဖြတ်မှု ဇယားကို နမူနာအဖြစ် အသုံးပြုပါ -

1. **ကိုယ်ပိုင် ရွှေ dataset** ကို သတ်မှတ်ပါ - မျှော်မှန်းသော keywords ပါဝင်သည့် စမ်းသပ်ချက် ၅ မှ ၁၀ ခုရေးပါ။
2. **ကိုယ်စားလှယ် prompt ကို ရေးပါ** - စမ်းသပ်လိုသော အသုံးပြုသူ ညွှန်ကြားချက်များရေးပါ။
3. **အကဲဖြတ်မှု လည်ပတ်ပါ** - အခြေခံ အမှတ်များရယူပါ။
4. **ပြုပြင်ထပ်မံပါ** - prompt ကို ပြင်ဆင်ပြီး ထပ်မံ လည်ပတ်ပါ။
5. **သတ်မှတ်ချက် တစ်ခု ရွေးပါ** - ဥပမာ "0.75 အတွင်း သင့်တင်ရန်မပြုရ" သတ်မှတ်ပါ။

### လေ့ကျင့်ခန်း 6 - Zava Editor ပုံစံနှင့် ချိတ်ဆက်ခြင်း

Zava Creative Writer ၏ Editor ကိုယ်စားလှယ် (`zava-creative-writer-local/src/api/agents/editor/editor.py`) ကို ပြန်လည် ကြည့်ပါ -

```python
# အယ်ဒီတာသည် ထုတ်လုပ်မှုတွင် LLM-အဖြေရှာသူတစ်ဦး ဖြစ်ပါသည်။
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

ဤသည်မှာ အပိုင်း 8 တွင် အသုံးပြုသည့် LLM-အဖြစ် အကဲဖြတ်သူနဲ့ တူညီသော အယူအဆ ဖြစ်ပါသည်၊ သို့သော် offline စမ်းသပ်မှုစုစည်းမှုတွင် မဟုတ်ပဲ ထုတ်လုပ်မှု pipeline တွင် ထည့်သွင်းထားသည်။ နှစ်မျိုးလုံးမှာ -

- မော်ဒယ်မှ ဖွဲ့စည်းထားသော JSON output ကို အသုံးပြုသည်။
- စနစ် prompt တွင် သတ်မှတ်ထားသော အရည်အသွေး စံတော်ချိန်များကို လုပ်ဆောင်သည်။
- pass/fail ဆုံးဖြတ်ချက် ထုတ်သည်။

**ကွာခြားချက်**: Editor ကိုယ်စားလှယ်သည် ထုတ်လုပ်မှုတွင် (အမိန့်တိုင်းတွင်) လည်ပတ်သည်။ အကဲဖြတ်ဇယားသည် ဖွံ့ဖြိုးတိုးတက်မှုအတွင်း (သင် သယ်ဆောင်မတိုင်ခင်) လည်ပတ်သည်။

| **ပို့ဆောင်မှီ အကဲဖြတ်မှု** | အရည်အသွေး စံချိန်များ သတ်မှတ်ပြီး အကဲဖြတ်အမှတ်များဖြင့် ထုတ်ပေးမှုများအား လုံခြုံစွာ ထိန်းသိမ်းသည် |

## နောက်ထပ် လုပ်ဆောင်ရန်

- **တိုးချဲ့ပါ**: ရွှေ dataset နှင့် အမျိုးအစား များ ပိုမို ထည့်ပါ။
- **အလိုအလျောက်လုပ်ဆောင်ပါ**: Evaluation ကို CI/CD pipeline တွင် ပေါင်းစပ် ထည့်သွင်းပါ။
- **စွမ်းဆောင်ရည်မြင့် Judge များ**: သေးငယ်သော မော်ဒယ် ထုတ်ကုန်ကို စမ်းသပ်ရန် ကြီးမားသော မော်ဒယ်ကို Judge အဖြစ် အသုံးပြုပါ။
- **အချိန်အလျားတိုက်ကြည့်ပါ**: အကဲဖြတ်ရလဒ်များ သိမ်းဆည်းပြီး prompt နှင့် မော်ဒယ် ဗားရှင်းများ နှိုင်းယှဉ်နိုင်စေရန်။
