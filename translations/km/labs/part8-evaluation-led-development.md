## យល់ដឹងសំខាន់ៗ

### 1. ឯកសារមាស

**ឯកសារមាស** គឺជាសំណុំគំរូតេស្តដែលបានរៀបចំជាមុនជាមួយលទ្ធផលដែលរង់ចាំ។ គំរូតេស្តនីមួយៗរួមមាន៖

- **បញ្ចូល**៖ ជំនួសឬសំណួរដើម្បីផ្ញើទៅភ្នាក់ងារ
- **លទ្ធផលដែលរង់ចាំ**៖ អ្វីដែលការឆ្លើយតបត្រឹមត្រូវ ឬមានគុណភាពខ្ពស់គួរតែមាន (ពាក្យគន្លឹះ, រចនាសម្ព័ន្ធ, បញ្ចាក់)
- **ក្រុម**៖ ការប្រមូលផ្តុំសម្រាប់របាយការណ៍ (ឧ. "ភាពត្រឹមត្រូវនៃព័ត៌មាន","សម្លេង","ភាពពេញលេញ")

### 2. ការត្រួតពិនិត្យដោយច្បាប់

ការត្រួតពិនិត្យដ៏លឿន ពីលក្ខណៈយ៉ាងប្រាកដមិនត្រូវការរូបមន្ត LLM ៖

| ត្រួតពិនិត្យ | វាពិនិត្យអ្វី |
|-------|--------------|
| **កំណត់ប្រវែង** | ចម្លើយមិនខ្លីពេក (សេដ្ឋកិច្ច) ឬវែងពេក (ឡើងរឿង) |
| **ពាក្យគន្លឹះចាំបាច់** | ចម្លើយមានពាក្យ ឬអត្ថបទដែលរំពឹងទុក |
| **ការផ្ទៀងផ្ទាត់ទ្រង់ទ្រាយ** | JSON អាចបង្ហាញបាន, ចំណងជើង Markdown មានលក្ខណៈបង្ហាញ |
| **មាតិកាត្រូវហាមឃាត់** | មិនមានឈ្មោះម៉ាកនិយាយបញ្ជាក់ក្លែងក្លាយ, មិនមានការប្រកួតប្រជែង |

### 3. LLM ជាអ្នកតុលាការ

ប្រើ **ម៉ូដែលក្នុងស្រុកដូចគ្នា** ដើម្បីផ្ដល់ពិន្ទុខ្លួនឯង (ឬលទ្ធផលពីបង្អួចជំនួសផ្សេងៗ)។ អ្នកតុលាការទទួល:

- សំណួរដើម
- ចម្លើយភ្នាក់ងារ
- តម្រូវការវាយតម្លៃ

ហើយត្រឡប់ពិន្ទុដែលមានរចនាសម្ព័ន្ធ។ វាស្រដៀងនឹងរបៀប Editor ពីផ្នែក 7 តែអនុវត្តយ៉ាងប្រព្រឹត្តទៅលើសំណុំតេស្ត។

### ការហាត់ 3 - យល់ដឹងពីការផ្ដល់ពិន្ទុ Rule-Based ជាមួយ LLM ជាអ្នកតុលាការ

សៀវភៅបួសការវាយតម្លៃប្រើ **ស្រទាប់ពិន្ទុពីរមុខ** ៖

#### ការត្រួតពិនិត្យដោយច្បាប់ (លឿន, ប្រាកដ)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM ជាអ្នកតុលាការ (មានលក្ខណៈជាប់ស្និទ្ធ, គុណភាព)

ម៉ូដែលក្នុងស្រុកដូចគ្នាមានតួនាទីជាអ្នកតុលាការជាមួយបន្ទាត់ក្រមរៀបចំ៖

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**សំណួរដើម្បីគិត៖**
1. ពេលណាអ្នកនឹងទុកចិត្តត្រួតពិនិត្យដោយច្បាប់ជាង LLM ជាអ្នកតុលាការ?
2. តើម៉ូដែលអាចវាយតម្លៃលទ្ធផលរបស់ខ្លួនបានយ៉ាងទៀងទាត់មែនទេ? មានកំណត់អ្វីខ្លះ?
3. វាប្រៀបធៀបយ៉ាងដូចម្តេចជាមួយលំនាំភ្នាក់ងារ Editor ពីផ្នែក 7?

### ការហាត់ 4 - ប្រៀបធៀបជំនួស Prompt

ឧទាហរណ៍នេះរត់ **បញ្ចូលជំនួសពីរគ្នា** លើគំរូតេស្តដូចគ្នា៖

| ជំនួស | ស្ទីល Prompt ប្រព័ន្ធ |
|---------|-------------------|
| **មូលដ្ឋាន** | ធម្មតា៖ "អ្នកជាជំនួយការជួយទៀងទាត់" |
| **ពិសេស** | ពិពណ៌នាលម្អិត៖ "អ្នកជាអ្នកជំនាញ Zava DIY ដែលផ្ដល់អនុសាសន៍ផលិតផលជាក់លាក់ និងផ្ដល់មុខទំនាក់ទំនងជំហាន-ជំហាន" |

បន្ទាប់ពីរត់ អ្នកនឹងឃើញកាតពិន្ទូច្រើនដូចជាៈ

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**លំហាត់៖**
1. រត់ការវាយតម្លៃ និងកត់សំគាល់ពិន្ទុសម្រាប់ជំនួសនីមួយៗ
2. ប្ដូរជំនួសពិសេសឱ្យកាន់តែលម្អិត។ តើពិន្ទុខ្លះល្អឡើងទេ?
3. បន្ថែមជំនួស prompt ទីបី ហើយប្រៀបធៀបទាំងបី។
4. ព្យាយាមប្ដូរឈ្មោះម៉ូដែល (ឧ. `phi-4-mini` ប្រឆាំង `phi-3.5-mini`) ហើយប្រៀបធៀបលទ្ធផល។

### ការហាត់ 5 - អនុវត្តការវាយតម្លៃទៅភ្នាក់ងាររបស់អ្នក

ប្រើសៀវភៅបួសនេះជាគំរូសម្រាប់ភ្នាក់ងាររបស់អ្នក៖

1. **កំណត់ឯកសារមាសរបស់អ្នក**៖ សរសេរករណីតេស្ត 5 ដល់ 10 ជាមួយពាក្យគន្លឹះដែលរង់ចាំ។
2. **សរសេរ prompt ប្រព័ន្ធរបស់អ្នក**៖ ជាមួយនឹងឱកាសដែលអ្នកចង់តេស្ត។
3. **រត់ការវាយតម្លៃ**៖ ទទួលពិន្ទុមូលដ្ឋាន។
4. **ធ្វើការកែប្រែ**៖ ប្ដូរ prompt, រត់ម្ដងទៀត, និងប្រៀបធៀប។
5. **កំណត់ហ្រ្វេប្រាក់**៖ ឧ. "មិនចែកចាយក្រោមពិន្ទុរួម 0.75"។

### ការហាត់ 6 - ការតភ្ជាប់ទៅលំនាំ Zava Editor

មើលទៅវិញទៅមកភ្នាក់ងារ Editor របស់ Zava Creative Writer (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# អ្នកកែសម្រួលគឺជាអ្នកវាយតម្លៃ LLM ក្នុងផលិតកម្ម៖
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

នេះគឺជាគំនិតដូចគ្នានៃ LLM ជាអ្នកតុលាការពីផ្នែក 8 ប៉ុន្តែបានបញ្ចូលក្នុងប៉ាយប៊ែនផលិតភាពជាជ្រួលជ្រើស ផ្ទុយពីសំណុំតេស្តក្រៅបណ្ដាញ។ លំនាំទាំងពីរនេះ៖

- ប្រើលទ្ធផល JSON ដែលមានរចនាសម្ព័ន្ធពីម៉ូដែល។
- អនុវត្តលក្ខខណ្ឌគុណភាពដែលបានកំណត់ក្នុង prompt ប្រព័ន្ធ។
- សម្រេចចិត្តជាប់/បរាជ័យ។

**ភាពខុសគ្នា៖** អ្នកនិពន្ធដំណើរការនៅក្នុងផលិតភាព (លើសំណើរ ទាំងអស់)។ សៀវភៅបួសការវាយតម្លៃដំណើរការនៅពេលអភិវឌ្ឍ (មុនចែកចាយ)។

## ជំហានបន្ទាប់

- **ពង្រីក**៖ បន្ថែមគំរូតេស្ត និងក្រុមទៅឯកសារមាសរបស់អ្នក។
- **ស្វ័យប្រវត្តិកម្ម**៖ បញ្ចូលការវាយតម្លៃក្នុងបណ្ដាញ CI/CD របស់អ្នក។
- **អ្នកតុលាការវិជ្ជាជីវៈ**៖ ប្រើម៉ូដែលធំជាអ្នកតុលាការពេលតេស្តលទ្ធផលម៉ូដែលតូច។
- **តាមដានពេលវេលា**៖ រក្សាទុកលទ្ធផលវាយតម្លៃសម្រាប់ប្រៀបធៀបជាមួយប៊ិច prompt និងម៉ូដែល។
