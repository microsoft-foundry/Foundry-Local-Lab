# الجزء 8: تطوير بقيادة التقييم مع Foundry Local

> **الهدف:** بناء إطار عمل للتقييم يختبر ويقيم وكلاء الذكاء الاصطناعي لديك بشكل منهجي، باستخدام نفس النموذج المحلي ليكون كل من الوكيل تحت الاختبار والحكم، حتى تتمكن من تكرار المطالبات بثقة قبل الإطلاق.

## لماذا التطوير بقيادة التقييم؟

عند بناء وكلاء الذكاء الاصطناعي، فإن عبارة "يبدو أنه صحيح إلى حد ما" ليست كافية. **التطوير بقيادة التقييم** يعامل مخرجات الوكيل ككود برمجي: تكتب الاختبارات أولاً، وتقيس الجودة، وترسل فقط عندما تصل الدرجات إلى عتبة معينة.

في كاتب زافا الإبداعي (الجزء 7)، كان **وكيل المحرر** بالفعل يعمل كمقيم خفيف الوزن؛ يقرر القبول أو المراجعة. هذا المختبر يصيغ هذا النمط في إطار تقييم متكرر يمكنك تطبيقه على أي وكيل أو سير عمل.

| المشكلة | الحل |
|---------|----------|
| التغيرات في المطالبات تكسر الجودة بصمت | **مجموعة بيانات ذهبية** تكتشف التراجعات |
| التحيز لـ "يعمل على مثال واحد" | **حالات اختبار متعددة** تكشف الحالات الطرفية |
| تقييم الجودة ذاتيًا | **القاعدة + تقييم الحكم بواسطة LLM** توفر أرقامًا |
| لا طريقة لمقارنة متغيرات المطالبات | **تشغيل التقييم جنبًا إلى جنب** مع تجميع الدرجات |

---

## المفاهيم الأساسية

### 1. مجموعات البيانات الذهبية

**مجموعة البيانات الذهبية** هي مجموعة منتقاة من حالات الاختبار مع مخرجات متوقعة معروفة. كل حالة اختبار تشمل:

- **الإدخال**: المطالبة أو السؤال المرسل إلى الوكيل
- **المخرجات المتوقعة**: ما يجب أن تحتويه الاستجابة الصحيحة أو عالية الجودة (الكلمات المفتاحية، الهيكل، الوقائع)
- **الفئة**: التجميع للتقارير (مثل "الدقة الواقعية"، "النبرة"، "الشمولية")

### 2. الفحوصات القائمة على القواعد

فحوصات سريعة وحتمية لا تتطلب نموذج لغة كبير:

| الفحص | ما يختبره |
|-------|-----------|
| **حدود الطول** | الاستجابة ليست قصيرة جدًا (كسولة) أو طويلة جدًا (متشعبة) |
| **الكلمات المفتاحية المطلوبة** | الاستجابة تذكر المصطلحات أو الكيانات المتوقعة |
| **التحقق من الصيغة** | JSON قابل للتحليل، ورؤوس Markdown موجودة |
| **المحتوى الممنوع** | لا أسماء علامات تجارية متخيلة، ولا إشارات إلى المنافسين |

### 3. الحكم بواسطة LLM

استخدم **نفس النموذج المحلي** لتقييم مخرجاته الخاصة (أو مخرجات من متغير مطالبة مختلف). يتلقى الحكم:

- السؤال الأصلي
- استجابة الوكيل
- معايير التقييم

ويرجع درجة منظمة. هذا يعكس نمط المحرر من الجزء 7 لكن يُطبق بشكل منهجي عبر مجموعة اختبار.

### التمرين 2 - فهم مجموعة البيانات الذهبية

استعرض حالات الاختبار المعرفة في نموذج التقييم. كل حالة اختبار تحتوي على:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**أسئلة للنظر:**
1. لماذا القيم المتوقعة هي **كلمات مفتاحية** بدلاً من جمل كاملة؟
2. كم عدد حالات الاختبار التي تحتاجها لتقييم موثوق؟
3. ما الفئات التي ستضيفها لتطبيقك الخاص؟

---

### التمرين 3 - فهم تقييم القاعدة مقابل الحكم بواسطة LLM

يستخدم إطار التقييم **طبقتين من التقييم**:

#### الفحوصات القائمة على القواعد (سريعة، حتمية)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### الحكم باستخدام LLM (متعمق، نوعي)

يعمل نفس النموذج المحلي كحكم مع معايير منظمة:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**أسئلة للنظر:**
1. متى تثق بالفحوصات القائمة على القواعد أكثر من الحكم بواسطة LLM؟
2. هل يمكن للنموذج أن يقيم مخرجاته بشكل موثوق؟ ما هي القيود؟
3. كيف يقارن هذا مع نمط وكيل المحرر في الجزء 7؟

---

### التمرين 4 - مقارنة متغيرات المطالبة

يشغل النموذج عينة **متغيري مطالبة** ضد نفس حالات الاختبار:

| المتغير | أسلوب مطالبة النظام |
|---------|--------------------|
| **القيمة الأساسية** | عام: "أنت مساعد مفيد" |
| **المتخصص** | مفصل: "أنت خبير Zava DIY يوصي بمنتجات محددة ويوفر إرشادات خطوة بخطوة" |

بعد التشغيل سترى بطاقة درجات مثل:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**التمارين:**
1. شغّل التقييم ودوّن الدرجات لكل متغير
2. عدّل المطالبة المتخصصة لتكون أكثر تحديدًا. هل تتحسن الدرجة؟
3. أضف متغير مطالبة ثالث وقارن الثلاثة جميعًا.
4. جرّب تغيير اسم النموذج (مثل `phi-4-mini` مقابل `phi-3.5-mini`) وقارن النتائج.

---

### التمرين 5 - تطبيق التقييم على وكيلك الخاص

استخدم إطار التقييم كنموذج لوكلائك:

1. **حدد مجموعة البيانات الذهبية الخاصة بك**: اكتب 5 إلى 10 حالات اختبار بكلمات مفتاحية متوقعة.
2. **اكتب مطالبة نظامك**: تعليمات الوكيل التي تريد اختبارها.
3. **شغّل التقييم**: احصل على الدرجات الأساسية.
4. **كرر**: عدّل المطالبة، شغل مرة أخرى، وقارن.
5. **حدد عتبة**: مثل "لا ترسل أقل من درجة مركبة 0.75".

---

### التمرين 6 - الربط بنمط محرر زافا

أعد النظر في وكيل المحرر في Zava Creative Writer (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# المُحرِّر هو نموذج لغة كبير كقاضٍ في الإنتاج:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

هذا هو **نفس المفهوم** كجزء 8 الحكم بواسطة LLM، لكنه مدمج في خط الإنتاج بدلاً من مجموعة اختبار غير متصلة. كلا النمطين:

- يستخدمان مخرجات JSON منظمة من النموذج.
- يطبقان معايير الجودة المعرفة في مطالبة النظام.
- يتخذان قرار نجاح أو فشل.

**الفرق:** يعمل المحرر في الإنتاج (على كل طلب). إطار التقييم يعمل في التطوير (قبل الإطلاق).

## النقاط الرئيسية

| المفهوم | المأخذ |
|---------|---------|
| **مجموعات البيانات الذهبية** | انتقِ حالات اختبار مبكرًا؛ فهي اختبارات التراجع لوكيل الذكاء الاصطناعي |
| **الفحوصات القائمة على القواعد** | سريعة، حتمية، وتكتشف الأخطاء الواضحة (الطول، الكلمات المفتاحية، الصيغة) |
| **الحكم بواسطة LLM** | تقييم جودة دقيق باستخدام نفس النموذج المحلي |
| **مقارنة المطالبات** | شغّل متغيرات متعددة ضد نفس مجموعة الاختبار لاختيار الأفضل |
| **الميزة على الجهاز** | كل التقييمات تتم محليًا: لا تكاليف API، لا حدود معدلات، لا خروج بيانات من جهازك |
| **التقييم قبل الإطلاق** | حدّد عتبات جودة واعتمدها لإطلاق الإصدارات بناءً على نتائج التقييم |
