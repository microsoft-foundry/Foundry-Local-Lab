# भाग ८: Foundry Local सहित मूल्यांकन-केंद्रित विकास

> **लक्ष्य:** मूल्यांकन फ्रेमवर्क निर्माण गर्नुहोस् जसले तपाईंका AI एजेन्टहरूलाई व्यवस्थित रूपमा परीक्षण र स्कोर गर्दछ, त्यसै स्थानीय मोडेललाई परीक्षण अन्तर्गत एजेन्ट र निर्णायक दुबैको रूपमा प्रयोग गरेर, ताकि तपाईंले विश्वस्त भएर प्रॉम्प्टहरू पुनरावृत्ति गर्न सक्नुहुनेछ।

## किन मूल्यांकन-केंद्रित विकास?

AI एजेन्टहरू बनाउँदा, "यो ठीक जस्तो देखिन्छ" पर्याप्त छैन। **मूल्यांकन-केंद्रित विकास** एजेन्टका आउटपुटहरूलाई कोडजस्तो व्यवहार गर्छ: तपाईंले पहिले परीक्षणहरू लेख्नुहुन्छ, गुणस्तर मापन गर्नुहुन्छ, र स्कोरहरूले थ्रेसहोल्ड पूरा गर्दा मात्र पठाउनुहुन्छ।

Zava Creative Writer (भाग ७) मा, **संपादक एजेन्ट** पहिले नै एक हल्का मूल्यांकनकर्ता को रूपमा काम गर्छ; यसले ACCEPT वा REVISE निर्णय गर्छ। यो प्रयोगशाला त्यो ढाँचालाई औपचारिक रूपमा पुनरावृत्ति गर्न सकिने मूल्यांकन फ्रेमवर्कमा परिणत गर्छ जुन तपाईं कुनै पनि एजेन्ट वा पाइपलाइनमा लागू गर्न सक्नुहुन्छ।

| समस्या | समाधान |
|---------|----------|
| प्रॉम्प्ट परिवर्तनहरूले मौन रूपमा गुणस्तर भंग गर्छन् | **गोल्डेन डेटासेट** ले पुनरागमनहरू समात्छ |
| "एउटा उदाहरणमा काम गर्छ" पक्षपात | **धेरै परीक्षण केसहरू** ले किनारका मामिलाहरू देखाउँछन् |
| विषयगत गुणस्तर मूल्यांकन | **नियम-आधारित + LLM-जज स्कोरिंग** ले संख्याहरू प्रदान गर्छ |
| प्रॉम्प्ट भेरियंटहरूको तुलना गर्ने कुनै तरिका छैन | **साइड-बाइ-साइड मूल्यांकन दौडहरू** संग समग्र स्कोरहरू |

## प्रमुख अवधारणाहरू

### १. गोल्डेन डेटासेटहरू

**गोल्डेन डेटासेट** भनेको पूर्वनिर्धारित अपेक्षित आउटपुटहरू सहितको संरचित परीक्षण केसहरूको सेट हो। प्रत्येक परीक्षण केसले समावेश गर्दछ:

- **इनपुट**: एजेन्टलाई पठाउनुपर्ने प्रॉम्प्ट वा प्रश्न
- **अपेक्षित आउटपुट**: सही वा उच्च गुणस्तरको प्रतिक्रिया के हुनु पर्छ (कीवर्डहरू, संरचना, तथ्यहरू)
- **श्रेणी**: रिपोर्टिङका लागि वर्गीकरण (जस्तै "तथ्यिक शुद्धता", "स्वर", "पूर्णता")

### २. नियम-आधारित जाँचहरू

द्रुत, निर्धारक जाँचहरू जुन LLM आवश्यक पर्दैनन्:

| जाँच | के परीक्षण गर्छ |
|-------|--------------|
| **लम्बाई सीमा** | प्रतिक्रिया धेरै छोटो (आलसी) वा धेरै लामो (फिर्ता लाग्ने) छैन |
| **आवश्यक कीवर्डहरू** | प्रतिक्रियाले अपेक्षित शब्द वा संस्थाहरू उल्लेख गर्छ |
| **ढाँचा प्रमाणिकरण** | JSON पार्स गर्न मिल्ने, Markdown शीर्षकहरू उपस्थित छन् |
| **निषिद्ध सामग्री** | काल्पनिक ब्रान्ड नामहरू छैनन्, प्रतिस्पर्धी उल्लेख छैन |

### ३. LLM-जज

आफ्नोै आउटपुट (वा फरक प्रॉम्प्ट भेरियन्टको आउटपुट) को मूल्याङ्कन गर्न समान स्थानीय मोडेल प्रयोग गर्नुहोस्। निर्णायकले प्राप्त गर्छ:

- मूल प्रश्न
- एजेन्टको प्रतिक्रिया
- मूल्याङ्कन मापदण्डहरू

र संरचित स्कोर फर्काउँछ। यो पार्ट ७ को सम्पादक ढाँचाको समान छ तर परीक्षण सूटमा प्रणालीगत रूपमा लागू गरिएको छ।

### अभ्यास २ - गोल्डेन डेटासेट बुझ्नुहोस्

मूल्यांकन नमूनामा परिभाषित परीक्षण केसहरू जाँच्नुहोस्। प्रत्येक परीक्षण केससँग छ:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**विचार गर्नका लागि प्रश्नहरू:**
1. किन अपेक्षित मानहरू पूर्ण वाक्यभन्दा **कीवर्डहरू** छन्?
2. भरोसायोग्य मूल्याङ्कनका लागि तपाईंलाई कति परीक्षण केसहरू चाहिन्छ?
3. तपाईंको आफ्नै अनुप्रयोगको लागि कुन वर्गहरू थप्नुहुन्छ?

---

### अभ्यास ३ - नियम-आधारित र LLM-जज स्कोरिंग बुझ्नुहोस्

मूल्यांकन फ्रेमवर्कले **दुई स्कोरिङ स्तरहरू** प्रयोग गर्छ:

#### नियम-आधारित जाँचहरू (द्रुत, निर्धारक)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-जज (सूक्ष्म, गुणात्मक)

त्यही स्थानीय मोडेल एक निर्णायकको रूपमा काम गर्छ, संरचित रूब्रिकसहित:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**विचार गर्नका लागि प्रश्नहरू:**
1. कहिले नियम-आधारित जाँचहरूमा LLM-जज भन्दा विश्वास गर्नुहुन्छ?
2. के कुनै मोडेलले आफनो आउटपुट विश्वसनीय रूपमा निर्णय गर्न सक्छ? के सीमितताहरू छन्?
3. यो पार्ट ७ को संपादक एजेन्ट ढाँचासँग कसरी तुलना हुन्छ?

---

### अभ्यास ४ - प्रॉम्प्ट भेरियन्टहरूको तुलना गर्नुहोस्

नमूनाले एउटै परीक्षण केसहरूमा **दुई प्रॉम्प्ट भेरियन्टहरू** चलाउँछ:

| भेरियन्ट | प्रणाली प्रॉम्प्ट शैली |
|---------|-------------------|
| **आधारभूत** | सामान्य: "तपाईं एक सहायक सहायक हुनुहुन्छ" |
| **विशेषज्ञ** | विस्तृत: "तपाईं एक Zava DIY विशेषज्ञ हुनुहुन्छ जसले विशेष उत्पादनहरू सिफारिस गर्दछ र चरण-द्वारा-चरण मार्गदर्शन दिनुहुन्छ" |

चलाएपछि, तपाईंलाई स्कोरकार्ड देखिनेछ:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**अभ्यासहरू:**
1. मूल्यांकन चलाउनुहोस् र हरेक भेरियन्टको स्कोर नोट गर्नुहोस्
2. विशेषज्ञ प्रॉम्प्टलाई अझ विशेष बनाउनुहोस्। के स्कोर सुधारिन्छ?
3. तेस्रो प्रॉम्प्ट भेरियन्ट थप्नुहोस् र सबै तीन तुलना गर्नुहोस्।
4. मोडेल उपनाम परिवर्तन गरेर प्रयास गर्नुहोस् (जस्तै `phi-4-mini` विरुद्ध `phi-3.5-mini`) र नतिजा तुलना गर्नुहोस्।

---

### अभ्यास ५ - आफ्नो एजेन्टमा मूल्यांकन लागू गर्नुहोस्

तपाईंका आफ्नै एजेन्टका लागि मूल्यांकन फ्रेमवर्कलाई टेम्प्लेटको रूपमा प्रयोग गर्नुहोस्:

1. **आफ्नो गोल्डेन डेटासेट परिभाषित गर्नुहोस्**: ५ देखि १० परीक्षण केसहरू कीवर्डसहित लेख्नुहोस्।
2. **आफ्नो सिस्टम प्रॉम्प्ट लेख्नुहोस्**: परीक्षण गर्न चाहेको एजेन्ट निर्देशनहरू।
3. **मूल्यांकन चलाउनुहोस्**: आधारभूत स्कोरहरू प्राप्त गर्नुहोस्।
4. **पुनरावृत्ति गर्नुहोस्**: प्रॉम्प्ट परिमार्जन गर्नुहोस्, फेरि चलाउनुहोस्, र तुलना गर्नुहोस्।
5. **थ्रेसहोल्ड सेट गर्नुहोस्**: जस्तै "०.७५ संयुक्त स्कोर भन्दा तल पठाउनु हुँदैन"।

### अभ्यास ६ - Zava संपादक ढाँचासँग कनेक्शन

Zava Creative Writer को संपादक एजेन्ट हेर्नुहोस् (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# सम्पादक उत्पादनमा एक LLM-को रूपमा न्यायाधीश हो:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

यो **उही अवधारणा** हो जुन भाग ८ को LLM-जजसँग मेल खान्छ, तर उत्पादन पाइपलाइनमा एम्बेड गरिएको छ, न कि अफलाइन परीक्षण सूटमा। दुबै ढाँचाहरू:

- मोडेलबाट संरचित JSON आउटपुट प्रयोग गर्छन्।
- प्रणाली प्रॉम्प्टमा परिभाषित गुणस्तर मापदण्ड लागू गर्छन्।
- पास/फेल निर्णय गर्छन्।

**भिन्नता:** संपादक उत्पादनमा (हरेक अनुरोधमा) चल्छ। मूल्यांकन फ्रेमवर्क विकासमा (पठाउनु अघि) चल्छ।

## आगामी कदमहरू

- **स्केल अप गर्नुहोस्**: थप परीक्षण केसहरू र वर्गहरू तपाईंको गोल्डेन डेटासेटमा थप्नुहोस्।
- **स्वचालित गर्नुहोस्**: मूल्यांकनलाई तपाईंको CI/CD पाइपलाइनमा एकीकृत गर्नुहोस्।
- **उन्नत निर्णायकहरू**: ठूलो मोडेललाई निर्णायकको रूपमा प्रयोग गरी सानो मोडेलको आउटपुट परीक्षण गर्नुहोस्।
- **समय अनुसार ट्रयाक गर्नुहोस्**: मूल्यांकन नतिजाहरू बचत गर्नुहोस् र प्रॉम्प्ट तथा मोडेल भर्सनहरू बीच तुलना गर्नुहोस्।
