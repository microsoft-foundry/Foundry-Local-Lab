# חלק 8: פיתוח מונחה הערכה עם Foundry Local

> **מטרה:** לבנות מסגרת הערכה שבאופן שיטתי בודקת ומדרגת את סוכני ה-AI שלך, תוך שימוש באותו מודל מקומי הן כסוכן הנבדק והן כשופט, כדי שתוכל לחזור על ניסוחים בביטחון לפני משלוח.

## מדוע פיתוח מונחה הערכה?

כשבונים סוכני AI, "נראה בסדר" זה לא מספיק. **פיתוח מונחה הערכה** מתייחס לפלטים של הסוכן כמו לקוד: קודם כותבים בדיקות, מודדים איכות, ומשלחים רק כשהציונים מגיעים לסף שנקבע.

ב-Zava Creative Writer (חלק 7), **סוכן העורך** כבר מתפקד כמעריך קל; הוא מחליט האם לקבל או לתקן. המעבדה הזו מחדדת את הדפוס הזה למסגרת הערכה שניתן לחזור עליה וליישם על כל סוכן או תהליך.

| בעיה | פתרון |
|---------|----------|
| שינויים בשאילתא שוברים איכות בשקט | **מקור נתונים זהב** תופס נסיגות |
| הטיית "עובד על דוגמה אחת" | **מבחני מקרה מרובים** חושפים מקרי קצה |
| הערכת איכות סובייקטיבית | **דרוג מבוסס כללים + LLM כשופט** מספק מספרים |
| אין דרך להשוות וריאציות של הנחיות | **הרצות הערכה מקבילות** עם צבירת ציונים |

## מושגים מרכזיים

### 1. מקורות נתונים זהב

**מקור נתונים זהב** הוא אוסף ממויין של מקרים לבדיקה עם פלטים צפויים ידועים. כל מקרה בדיקה כולל:

- **קלט**: השאילתא או השאלה לשלוח לסוכן
- **פלט צפוי**: מה שתגובה נכונה או איכותית צריכה לכלול (מילות מפתח, מבנה, עובדות)
- **קטגוריה**: קיבוץ לדווח (למשל "דיוק עובדתי", "טון", "שלמות")

### 2. בדיקות מבוססות כללים

בדיקות מהירות ודטרמיניסטיות שאינן דורשות LLM:

| בדיקה | מה היא בודקת |
|-------|--------------|
| **גבולות אורך** | התגובה לא קצרה מדי (עצלנית) ולא ארוכה מדי (מתרחקת מהנושא) |
| **מילות מפתח נדרשות** | התגובה מזכירה מונחים או ישויות צפויות |
| **אימות פורמט** | JSON ניתן לפרסור, כותרות Markdown נוכחות |
| **תוכן אסור** | אין שמות מותגים מופרכים, אין אזכורים למתחרים |

### 3. LLM כשופט

השתמש ב**אותו מודל מקומי** כדי לדרג את הפלטים שלו עצמו (או את הפלטים מסוגר שאילתה שונה). השופט מקבל:

- השאלה המקורית
- תגובת הסוכן
- קריטריוני דירוג

ומחזיר ציון מובנה. זה משקף את דפוס העורך מחלק 7 אך מיושם באופן שיטתי על פני מערך בדיקה.

### תרגיל 2 - הבנת מקור הנתונים הזהב

בדוק את מקרים הבדיקה שהוגדרו בדוגמת ההערכה. כל מקרה בדיקה כולל:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**שאלות למחשבה:**
1. מדוע הערכים הצפויים הם **מילות מפתח** במקום משפטים מלאים?
2. כמה מקרים בדיקה דרושים להערכה אמינה?
3. אילו קטגוריות היית מוסיף ליישום שלך?

---

### תרגיל 3 - הבנת דרוג מבוסס כללים מול LLM כשופט

מסגרת ההערכה משתמשת ב**שני רכיבי דרוג**:

#### בדיקות מבוססות כללים (מהירות, דטרמיניסטיות)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM כשופט (מעודן, איכותי)

אותו מודל מקומי משמש כשופט עם טבלה מובנית:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**שאלות למחשבה:**
1. מתי תסמוך על בדיקות מבוססות כללים על פני LLM כשופט?
2. האם מודל יכול להעריך באופן אמין את הפלט שלו עצמו? מהן המגבלות?
3. איך זה משתווה לדפוס סוכן העורך מחלק 7?

---

### תרגיל 4 - השוואת וריאציות שאילתא

הדוגמה מריצה **שתי וריאציות שאילתא** על אותם מקרים בדיקה:

| וריאציה | סגנון הנחיית מערכת |
|---------|-------------------|
| **בסיסית** | כללי: "אתה עוזר מועיל" |
| **מתמחה** | מפורט: "אתה מומחה Zava DIY שממליץ על מוצרים ספציפיים ומספק הדרכה שלב-אחר-שלב" |

לאחר ההרצה תראה כרטיס ציונים כגון:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**תרגילים:**
1. הרץ את ההערכה ורשום את הציונים לכל וריאציה
2. שנה את ההנחיה המתמחה להיות אפילו יותר ספציפית. האם הציון משתפר?
3. הוסף וריאציית הנחיה שלישית והשווה את השלוש.
4. נסה לשנות את כינוי המודל (למשל `phi-4-mini` לעומת `phi-3.5-mini`) והשווה תוצאות.

---

### תרגיל 5 - החל הערכה על הסוכן שלך

השתמש במסגרת ההערכה כתבנית לסוכנים שלך:

1. **הגדר את מקור הנתונים הזהב שלך**: כתוב 5 עד 10 מקרים עם מילות מפתח צפויות.
2. **כתוב את הנחיית המערכת שלך**: הוראות הסוכן שברצונך לבדוק.
3. **הרץ את ההערכה**: קבל ציוני בסיס.
4. **חזור על התהליך**: שפר את ההנחיה, הרץ שוב והשווה.
5. **קבע סף**: למשל "אל תשלח מתחת לציון משולב של 0.75".

### תרגיל 6 - חיבור לדפוס העורך של Zava

הביט חזרה על סוכן העורך של Zava Creative Writer (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# העורך הוא LLM כשופט בפרודקשן:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

זהו **אותו מושג** כמו LLM כשופט מהחלק 8, אך מוטמע בצנרת הייצור ולא במערך בדיקה לא מקוון. שני הדפוסים:

- משתמשים בפלט JSON מובנה מהמודל.
- מיישמים קריטריוני איכות שהוגדרו בהנחיית המערכת.
- מקבלים החלטת עבר/כישלון.

**השוני:** העורך פועל בייצור (בכל בקשה). מסגרת ההערכה פועלת בפיתוח (לפני משלוח).

## נקודות מרכזיות

| מושג | מסקנה |
|---------|----------|
| **מקורות נתונים זהב** | סווג מוקדם מקרים לבדיקה; הם בדיקות הרגרסיה שלך ל-AI |
| **בדיקות מבוססות כללים** | מהירות, דטרמיניסטיות, ותופסות כשלים ברורים (אורך, מילות מפתח, פורמט) |
| **LLM כשופט** | דירוג איכות מעודן באמצעות אותו מודל מקומי |
| **השוואת הנחיות** | הרץ וריאציות מרובות על אותו מערך בדיקה לבחור את הטובה ביותר |
| **יתרון מקומי** | כל הרצות ההערכה מתבצעות מקומית: ללא עלויות API, ללא מגבלות קצב, ללא יציאת נתונים מהמחשב שלך |
| **הערכה לפני משלוח** | קבע ספי איכות וחסום שחרור מתחת לציונים מסוימים |

## צעדים הבאים

- **הרחב**: הוסף מקרים וקטגוריות למקור הנתונים הזהב שלך.
- **אוטומט**: שלב הערכה בצנרת CI/CD שלך.
- **שופטים מתקדמים**: השתמש במודל גדול יותר כשופט בזמן שבודקים פלט של מודל קטן יותר.
- **עקוב לאורך זמן**: שמור תוצאות הערכה להשוואה בין וריאציות שאילתה וגרסאות מודל.
