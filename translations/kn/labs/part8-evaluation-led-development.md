# ಭಾಗ 8: Foundry Local ಸಹಿತ ಮೌಲ್ಯಮಾಪನ ಆಧಾರಿತ ಅಭಿವೃದ್ಧಿ

> **ಗೋಲು:** ಸಂಯೋಜಿತವಾಗಿ ನಿಮ್ಮ AI ಏಜೆಂಟ್‌ಗಳನ್ನು ಪರೀಕ್ಷಿಸುವ ಮತ್ತು ಅಂಕನೀಯ ಮಾಡುವ ಮೌಲ್ಯಮಾಪನ ರೂಪರೇಖೆಯನ್ನು ನಿರ್ಮಿಸಬೇಕು, ಪರೀಕ್ಷೆಯಡಿಯಲ್ಲಿ ಏಜೆಂಟ್ ಮತ್ತು ನ್ಯಾಯಮೂರ್ತಿಯಾಗಿ ಒಂದೇ ಸ್ಥಳೀಯ ಮಾದರಿಯನ್ನು ಬಳಸಿ, ಹೀಗಾಗಿ ನೀವು ಶಿಪ್ ಮಾಡುವ ಮೊದಲು ಪ್ರಾಂಪ್ಟ್‌ಗಳ ಮೇಲೆ ವಿಶ್ವಾಸದಿಂದ ಪುನರಾವರ್ತಿಸಬಹುದು.

## ಮೌಲ್ಯಮಾಪನ ಆಧಾರಿತ ಅಭಿವೃದ್ಧಿ ಏಕೆ?

AI ಏಜೆಂಟ್‌ಗಳನ್ನು ನಿರ್ಮಿಸುವಾಗ, "ಇದು ಸರಿಯಾದಂತೆ ಕಂಡುಬರುತ್ತದೆ" ಎಂದರೆ ಸಾಕಾಗದು. **ಮೌಲ್ಯಮಾಪನ ಆಧಾರಿತ ಅಭಿವೃದ್ಧಿ** ಏಜೆಂಟ್ ಔಟ್ಪುಟ್‌ಗಳನ್ನು ಕೋಡ್‌ಗಳಂತೆ ಹೆಚ್ಚಿಸುತ್ತದೆ: ನೀವು ಮೊದಲು ಪರೀಕ್ಷೆಗಳನ್ನು ಬರೆಯುತ್ತೀರಿ, ಗುಣಮಟ್ಟವನ್ನು ಅಳೆಯುತ್ತೀರಿ ಮತ್ತು ಅಂಕಗಳು ಗುರಿತ ಲೆಕ್ಕದೊಂದಿಗೆ ಬಂದಾಗಲೇ ಶಿಪ್ ಮಾಡುತ್ತೀರಿ.

Zava ಕ್ರಿಯೇಟಿವ್ ರೈಟರ್ (ಭಾಗ 7) ನಲ್ಲಿ, **ಎಡಿಟರ್ ಏಜೆಂಟ್** ಈಗಾಗಲೇ ಒಂದು ಕಡಿಮೆ ಭಾರವಾಗಿರೋ ಮೌಲ್ಯಮಾಪಕವಾಗಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ; ಅದು ಸ್ವೀಕರಿಸೋದು ಅಥವಾ ಮರುಪರಿಶೀಲನೆಯನ್ನು ನಿರ್ಧರಿಸುತ್ತದೆ. ಈ ಲ್ಯಾಬ್ ಆ ಮಾದರಿಯನ್ನು ಯಾವುದೇ ಏಜೆಂಟ್ ಅಥವಾ ಪೈಪ್ಲೈನ್‌ಗೆ ಅನ್ವಯಿಸಬಹುದಾದ ಪುನರಾವರ್ತಿಸಬಹುದಾದ ಮೌಲ್ಯಮಾಪನ ರೂಪರೇಖೆಯಾಗಿಸುತ್ತದೆ.

| ಸಮಸ್ಯೆ | ಪರಿಹಾರ |
|---------|----------|
| ಪ್ರಾಂಪ್ಟ್ ಬದಲಾವಣೆಗಳು ಮೌನವಾಗಿ ಗುಣಮಟ್ಟವನ್ನು ಸೇರಿದಂತೆ ತೊಂದರೆ ಉಂಟುಮಾಡುತ್ತವೆ | **ಗೋಲ್ಡ್ ಡೇಟಾಸೆಟ್** ಹಿಂಪಡೆಯುವುದನ್ನು ಪತ್ತೆಹಚ್ಚುತ್ತದೆ |
| "ಒಂದು ಉದಾಹರಣೆಯಲ್ಲಿ ಕೆಲಸ ಮಾಡುತ್ತದೆ" ಎಂಬ ಪಕ್ಕಬಾಧ್ಯತೆ | **ಬಹುಪಾಲು ಪರೀಕ್ಷಾ ಕೇಸ್‌ಗಳು** ಗಡಿಬಿಡಿ ಪ್ರಕರಣಗಳನ್ನು ಬಹಿರಂಗಪಡಿಸುತ್ತದೆ |
| ವೈಯಕ್ತಿಕ ಗುಣಮಟ್ಟ ಮೌಲ್ಯಮಾಪನ | **ನಿಯಮ ಆಧಾರಿತ + LLM-ನ್ಯಾಯಮೂರ್ತಿ ಅಂಕನ** ಸಂಖ್ಯೆಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ |
| ಪ್ರಾಂಪ್ಟ್ ಬೇರೆಯಾಂಶಗಳನ್ನು ಹೋಲಿಸುವ ಮಾರ್ಗವಿಲ್ಲ | **ಬದಿಯಲ್ಲಿ-ಬದಿ ಮೌಲ್ಯಮಾಪನ ಓಟಗಳು** ಸವಿವು ಕಾರ್ಯದ ಅಂಕಗಳನ್ನು ಒದಗಿಸುತ್ತದೆ |

## ಪ್ರಮುಖ ಕಲ್ಪನೆಗಳು

### 1. ಗೋಲ್ಡ್ ಡೇಟಾಸೆಟ್‌ಗಳು

**ಗೋಲ್ಡ್ ಡೇಟಾಸೆಟ್** ಒಂದು ತಿಳಿದ ನಿರೀಕ್ಷಿತ ಔಟ್ಪುಟ್‌ಗಳೊಂದಿಗೆ ಆಯ್ಕೆಮಾಡಲಾದ ಪರೀಕ್ಷಾ ಕೇಸ್‌ಗಳ ಸೇರ್ಪಡೆ. ಪ್ರತಿಯೊಂದು ಪರೀಕ್ಷಾ ಕೇಸ್ ಒಳಗೊಂಡಿದೆ:

- **ಇನ್ಪುಟ್:** ಏಜೆಂಟ್‌ಗೆ ಕಳುಹಿಸುವ ಪ್ರಾಂಪ್ಟ್ ಅಥವಾ ಪ್ರಶ್ನೆ
- **ನಿರೀಕ್ಷಿತ ಔಟ್ಪುಟ್:** ಸರಿಯಾದ ಅಥವಾ ಉನ್ನತ ಗುಣಮಟ್ಟದ ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ಇರಬೇಕಾದವುಗಳು (ಕೀವರ್ಡ್‌ಗಳು, ರಚನೆ, ವಾಸ್ತುಗಳ)
- **ಶ್ರೇಣಿ:** ವರದಿಗಾಗಿ ಗುಂಪು (ಉದಾ: "ವಾಸ್ತವಿಕ ಶುದ್ದತೆ", "ಟೋನ್", "ಪೂರ್ಣತೆ")

### 2. ನಿಯಮ ಆಧಾರಿತ ಪರಿಶೀಲನೆಗಳು

ವಿದೇಶಿ, ನಿರ್ಧಾರಾತ್ಮಕ ಪರಿಷೀಲನೆಗಳು ಯಾವಾಗಲೂ LLM ಅಗತ್ಯವಿಲ್ಲ:

| ಪರಿಶೀಲನೆ | ಏನು ಪರೀಕ್ಷಿಸುತ್ತದೆ |
|-------|--------------|
| **ನಿರಂತರತೆ ಬದಿಗಳು** | ಪ್ರತಿಕ್ರಿಯೆ ತುಂಬಾ ಚಿಕ್ಕದು (ಆಲಸ್ಯ) ಅಥವಾ ತುಂಬಾ ದೀರ್ಘ (ಗುಜುಗುಜು) ಅಲ್ಲ |
| **ಅಗತ್ಯವಾದ ಕೀವರ್ಡ್‌ಗಳು** | ಪ್ರತಿಕ್ರಿಯೆಯಲ್ಲಿ ನಿರೀಕ್ಷಿತ ಪದಗಳು ಅಥವಾ ಸತ್ತ್ವಗಳನ್ನು ಉಲ್ಲೇಖಿಸಬೇಕು |
| **ರೂಪ ಚೆಕ್** | JSON ಒಂದಿಗೋಡು, Markdown ಹೆಡರ್‌ಗಳು ಇದೆ |
| **ತಡೆ ಹೇರಿದ ವಿಷಯ** | ಯಾವುದೇ ಭ್ರಮೆಯಾದ ಬ್ರ್ಯಾಂಡ್ ಹೆಸರುಗಳು ಅಥವಾ ಸ್ಪರ್ಧಿ ಉಲ್ಲೇಖಗಳು ಬೇಡ |

### 3. LLM-ನೇಮಕಾತೀಯ ನ್ಯಾಯಮೂರ್ತಿ

ತಮ್ಮ ಸ್ವಂತ ಔಟ್ಪುಟ್‌ಗಳನ್ನು ಅಂಕನೀಯಗೊಳಿಸಲು (ಅಥವಾ ಬೇರೆ ಪ್ರಾಂಪ್ಟ್ ಬದಲಾವಣೆಯ ಔಟ್ಪುಟ್‌ಗಳಿಗೆ) **ಅದೇ ಸ್ಥಳೀಯ ಮಾದರಿಯನ್ನು** ಬಳಸಿ. ನ್ಯಾಯಮೂರ್ತಿಗೆ ಸಾಗುವದು:

- ಮೂಲ ಪ್ರಶ್ನೆ
- ಏಜೆಂಟ್‌ನ ಪ್ರತಿಕ್ರಿಯೆ
- ಅಂಕನೀಯ ಮಾಪಕ

ಒಂದು ರಚನೆಯಾದ ಅಂಕನ ನೀಡುತ್ತದೆ. ಇದು ಭಾಗ 7 ರ ಎಡಿಟರ್ ಮಾದರಿಯಂತೆ, ಆದರೆ ನಿರಂತರವಾಗಿ ಪರೀಕ್ಷಾ ಸರಣಿಯ ಮೇಲೆ ಅನ್ವಯಿಸುತ್ತದೆ.

### ವ್ಯಾಯಾಮ 2 - ಗೋಲ್ಡ್ ಡೇಟಾಸೆಟ್ ಅರ್ಥಮಾಡಿಕೊಂಡು

ಮೌಲ್ಯಮಾಪನ ಮಾದರಿಯಲ್ಲಿ ನಿರ್ಧಾರಗೊಂಡಿರುವ ಪರೀಕ್ಷಾ ಕೇಸುಗಳನ್ನು ಪರಿಶೀಲಿಸಿ. ಪ್ರತಿಯೊಂದು ಪರೀಕ್ಷಾ ಕೇಸ್ ಇದನ್ನು ಹೊಂದಿದೆ:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**ಪರಿಶೀಲಿಸುವ ಪ್ರಶ್ನೆಗಳು:**
1. ನಿರೀಕ್ಷಿತ ಮೌಲ್ಯಗಳು ಪೂರ್ಣ ವಾಕ್ಯಗಳಿಗಿಂತ **ಕೀವರ್ಡ್‌ಗಳು** ಆಗಿರುವ ಕಾರಣವೇನು?
2. ನಂಬಿಕಸ್ತರದ ಮೌಲ್ಯಮಾಪನಕ್ಕೆ ಎಷ್ಟು ಪರೀಕ್ಷಾ ಕೇಸ್ ಬೇಕಾಗುತ್ತದೆ?
3. ನಿಮ್ಮ ಸ್ವಂತ ಅನ್ವಯಿಕತೆಗೆ ಯಾವ ಶ್ರೇಣಿಗಳನ್ನು ಸೇರಿಸುವಿರಿ?

### ವ್ಯಾಯಾಮ 3 - ನಿಯಮ ಆಧಾರಿತ vs LLM-ನೇಮಕಾತೀಯ ನ್ಯಾಯಮೂರ್ತಿ ಅಂಕನ ಅರ್ಥಮಾಡಿಕೊಳ್ಳಿ

ಮೌಲ್ಯಮಾಪನ ರೂಪರೇಖೆ **ಎರಡು ಅಂಕನ ಮಟ್ಟಗಳನ್ನು** ಬಳಸುತ್ತದೆ:

#### ನಿಯಮ ಆಧಾರಿತ ಪರಿಶೀಲನೆಗಳು (ತ್ವರಿತ, ನಿರ್ಧಾರಾತ್ಮಕ)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-ನೇಮಕಾತೀಯ ನ್ಯಾಯಮೂರ್ತಿ (ಸೂಕ್ಷ್ಮ, ಗುಣಮಟ್ಟದ)

ಅದೇ ಸ್ಥಳೀಯ ಮಾದರಿ ಸಂಖ್ಯಾಪಟ್ಟಿಕೆಯನ್ನು ಹೊಂದಿರುವ ನ್ಯಾಯಮೂರ್ತಿಯಾಗಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**ಪರಿಶೀಲಿಸುವ ಪ್ರಶ್ನೆಗಳು:**
1. ನಿಯಮ ಆಧಾರಿತ ಪರಿಶೀಲನೆಗಳನ್ನು LLM-ನ್ಯಾಯಮೂರ್ತಿಗಿಂತ ಯಾವಾಗ ನಂಬಬಹುದು?
2. ಮಾದರಿ ತನ್ನ ಸ್ವಂತ ಔಟ್ಪುಟ್ ಅನ್ನು ನಂಬಿಕೆಯಿಂದ ನ್ಯಾಯಮೂರ್ತಿಯಾಗಿ ಅಂಕಮಾಡಬಹುದುವೇ? ಸೀಮಿತತೆಗಳು ಏನು?
3. ಇದು ಭಾಗ 7 ರ ಎಡಿಟರ್ ಏಜೆಂಟ್ ಮಾದರಿಯೊಂದಿಗೆ ಹೇಗೆ ಹೋಲುತ್ತದೆ?

---

### ವ್ಯಾಯಾಮ 4 - ಪ್ರಾಂಪ್ಟ್ ಬದಲಾವಣೆಗಳನ್ನು ಹೋಲಿಸಿ

ಉದಾಹರಣೆ **ಎರಡು ಪ್ರಾಂಪ್ಟ್ ಬದಲಾವಣೆಗಳನ್ನು** ಒಂದೇ ಪರೀಕ್ಷಾ ಕೇಸ್‌ಗಳಿಗೆ ಓಡಿಸುತ್ತದೆ:

| ಬದಲಾವಣೆ | ವ್ಯವಸ್ಥೆ ಪ್ರಾಂಪ್ಟ್ ಶೈಲಿ |
|---------|-------------------|
| **ಮೂಲ** | ಸಾಮಾನ್ಯ: "ನೀವು ಸಹಾಯ ಮಾಡುವ ಸಹಾಯಕರು" |
| **ವಿಶೇಷಿತ** | ವಿವರವಾದ: "ನೀವು ವಿಶೇಷ ಉತ್ಪನ್ನಗಳನ್ನು ಶಿಫಾರಸು ಮಾಡುವ ಮತ್ತು ಹಂತ-ಹಂತ ಮಾರ್ಗದರ್ಶನ ನೀಡುವ Zava DIY ತಜ್ಞರು" |

ಓಟದ ನಂತರ, ನೀವು ಹೀಗೆ ಅಂಕಪಟ್ಟಿಯನ್ನು ನೋಡಿರಬಹುದು:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**ವ್ಯಾಸಂಗಗಳು:**
1. ಮೌಲ್ಯಮಾಪನ ಓಡಿಸಿ ಪ್ರತಿಯೊಂದು ಬದಲಾವಣೆಗೆ ಅಂಕಗಳನ್ನು ಗಮನಿಸಿ
2. ವಿಶೇಷಿತ ಪ್ರಾಂಪ್ಟ್ ಅನ್ನು ಇನ್ನೂ ಹೆಚ್ಚು ವಿಶೇಷಗೊಳಿಸಿ. ಅಂಕಗಳು ಸುಧಾರಿಸುತ್ತವೆಯೇ?
3. ಮೂರನೇ ಪ್ರಾಂಪ್ಟ್ ಬದಲಾವಣೆಯನ್ನು ಸೇರಿಸಿ ಮೂರೂ ಹೋಲಿಸಿ.
4. ಮಾದರಿ ಪರ್ಯಾಯವನ್ನು (ಉದಾ: `phi-4-mini` ವಿರುದ್ಧ `phi-3.5-mini`) ಬದಲಿಸಿ ಫಲಿತಾಂಶಗಳನ್ನು ಹೋಲಿಸಿ.

---

### ವ್ಯಾಯಾಮ 5 - ನಿಮ್ಮ ಸ್ವಂತ ಏಜೆಂಟ್‌ಗೆ ಮೌಲ್ಯಮಾಪನ ಅನ್ವಯಿಸಿ

ನಿಮ್ಮ ಸ್ವಂತ ಏಜೆಂಟ್‌ಗಳಿಗೆ ಮಾದರಿಯಂತೆ ಮೌಲ್ಯಮಾಪನ ರೂಪರೇಖೆಯನ್ನು ಬಳಸಿಕೊಳ್ಳಿ:

1. **ನಿಮ್ಮ ಗೋಲ್ಡ್ ಡೇಟಾಸೆಟ್ ಅನ್ನು ರಚಿಸಿ:** ನಿರೀಕ್ಷಿತ ಕೀವರ್ಡ್‌ಗಳೊಂದಿಗೆ 5 ರಿಂದ 10 ಪರೀಕ್ಷಾ ಕೇಸ್ ಬರೆಯಿರಿ.
2. **ನಿಮ್ಮ ವ್ಯವಸ್ಥೆಯ ಪ್ರಾಂಪ್ಟ್ ಬರೆಯಿರಿ:** ಪರಿಶೀಲಿಸಲು ವಿಶೇಷ ಸೂಚನೆಗಳು.
3. **ಮೌಲ್ಯಮಾಪನ ಓಡಿಸಿ:** ಮೂಲ ಅಂಕಗಳನ್ನು ಪಡೆಯಿರಿ.
4. **ಪುನರಾವರ್ತನೆ:** ಪ್ರಾಂಪ್ಟ್ ಅನ್ನು ತಿದ್ದುಪಡಿ ಮಾಡಿ, ಮತ್ತೆ ಓಡಿಸಿ, ಮತ್ತು ಹೋಲಿಸಿ.
5. **ಮಿತಿ ನಿಗದಿಸು:** ಉದಾ. "0.75 ಸಂಯೋಜಿತ ಅಂಕಗಳ ಕೆಳಗೆ ಶಿಪ್ ಮಾಡಬೇಡಿ".

### ವ್ಯಾಯಾಮ 6 - Zava ಎಡಿಟರ್ ಮಾದರಿಯ ಸಂಪರ್ಕ

Zava Creative Writer ನ ಎಡಿಟರ್ ಏಜೆಂಟ್ (`zava-creative-writer-local/src/api/agents/editor/editor.py`) ಅನ್ನು ಹಿಂತಿರುಗಿ ನೋಡಿ:

```python
# ಸಂಪಾದಕರು ಉತ್ಪಾದನೆಯಲ್ಲಿ ನ್ಯಾಯಾಧಿಕಾರಿಯನ್ನು ಎಲ್ಎಲ್‌ಎಂಬಿ ಆಗಿದೆ:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

ಇದು ಭಾಗ 8 ರ LLM-ನೇಮಕಾತೀಯ ನ್ಯಾಯಮೂರ್ತಿ ಹಾಗೂ ಉತ್ಪಾದನಾ ಪೈಪ್ಲೈನ್‌ನಲ್ಲಿ ಅನ್ವಯಿಸುವ ಮೂಲಕ ಆಫ್‌ಲೈನ್ ಪರೀಕ್ಷಾ ಸರಣಿಯಿಂದ ಭಿನ್ನವಾಗಿದೆ. ಎರಡು ಮಾದರಿಗಳು:

- ಮಾದರಿಯಿಂದ ರಚನೆಯಾದ JSON ಔಟ್ಪುಟ್ ಬಳಸುತ್ತವೆ.
- ವ್ಯವಸ್ಥೆಯ ಪ್ರಾಂಪ್ಟ್‌ನಲ್ಲಿ ನಿರ್ಧರಿಸಲಾದ ಗುಣಮಟ್ಟ ಮಾನದಂಡಗಳನ್ನು ಅನ್ವಯಿಸುತ್ತವೆ.
- ಪಾಸ್/ಫೇಲ್ ನಿರ್ಧಾರ ತೆಗೆದುಕೊಳ್ಳುತ್ತವೆ.

**ಭಿನ್ನತೆ:** ಎಡಿಟರ್ ಉತ್ಪಾದನೆಯಲ್ಲಿ (ಪ್ರತಿ ವಿನಂತಿಗೆ) ಓಡುತ್ತದೆ. ಮೌಲ್ಯಮಾಪನ ರೂಪರೇಖೆ ಅಭಿವೃದ್ಧಿಯಲ್ಲಿ (ನೀವು ಶಿಪ್ ಮಾಡುವ ಮುಂಚೆ) ಓಡುತ್ತದೆ.

- **ಕಾಲಕಾಲದಲ್ಲಿ ಹಾದಿ ನೋಡಿ:** ಮೌಲ್ಯಮಾಪನ ಫಲಿತಾಂಶಗಳನ್ನು ಉಳಿಸಿ ಪ್ರತಿಯೊಂದು ಪ್ರಾಂಪ್ಟ್ ಮತ್ತು ಮಾದರಿ ಆವೃತ್ತಿಗಳ ನಡುವೆ ಹೋಲಿಸಿ.
