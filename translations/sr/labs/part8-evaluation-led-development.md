# Парт 8: Развој Вођен Проценом са Foundry Local

> **Циљ:** Изградити оквир за процену који систематски тестира и оцењује ваше AI агенте, користећи исти локални модел и као агента који се тестира и као судију, тако да можете са поверењем да усавршавате упите пре пуштања у рад.

## Зашто развој вођен пројеном?

Када правите AI агенте, „делује приближно исправно“ није довољно. **Развој вођен проценом** третира излазе агента као код: прво пишете тестове, мерите квалитет и шаљете у производњу тек када оцене достигну прагове.

У Zava Creative Writer-у (Парт 7), **Editор агент** већ делује као лагани оценитељ; он одлучује ПРИХВАТИ или ИСПРАВИ. Овај лабораторијски задатак формализује тај образац у поновљив оквир процене који можете применити на било ког агента или процес.

| Проблем | Решење |
|---------|---------|
| Промене упита тихо нарушавају квалитет | **Златни скуп података** открива регресије |
| Пристраност „ради на једном примеру“ | **Више тест случајева** открива ивичне случајеве |
| Субјективна процена квалитета | **Оцена базирана на правилима + LLM као судија** обезбеђује бројеве |
| Нема начина за поређење варијанти упита | **Паралелне процене** са агрегираним оценама |

## Кључни појмови

### 1. Златни скупови података

**Златни скуп података** је курирани скуп тест случајева са познатим очекиваним излазима. Сваки тест случај укључује:

- **Улаз**: Упит или питање којe се шаље агенту
- **Очекујeни излаз**: Шта исправан или квалитетан одговор треба да садржи (кључне речи, структуру, чињенице)
- **Категорија**: Груписање за извештавање (нпр. „чinjенична тачност“, „тон“, „потпуност“)

### 2. Провере базиране на правилима

Брзе, детерминистичке провере које не захтевају LLM:

| Провера | Шта тестира |
|---------|-------------|
| **Ограничења дужине** | Одговор није превише кратак (лепљив) или превише дугачак (лутање) |
| **Обавезне кључне речи** | Одговор помиње очекиване термине или ентитете |
| **Валидација формата** | JSON је парсиран, Markdown заглавља су присутна |
| **Забрањени садржај** | Није присутан халуцинирани назив бренда, нема помена конкурената |

### 3. LLM као судија

Користите **иста локални модел** да оцењује своје излазе (или излазе из друге варијанте упита). Судја добија:

- Оригинално питање
- Одговор агента
- Критеријуме оцењивања

И враћа структурирану оцену. Ово одражава образац Editora из Парт 7 али је применљиво систематски у целом скупу тестова.

### Вежба 2 - Разумевање златног скупа података

Погледајте дефинисане тест случајеве у примерку за процену. Сваки тест случај има:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**Питања за размишљање:**
1. Зашто су очекиване вредности **кључне речи**, а не целе реченице?
2. Колико тест случајева је потребно за поуздану процену?
3. Које категорије бисте додали за своју апликацију?

---

### Вежба 3 - Разумевање процене базиране на правилима и LLM као судија

Оквир за процену користи **два нивоа оцењивања**:

#### Провере базиране на правилима (брзе, детерминистичке)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM као судија (нијансирано, квалитативно)

Исти локални модел делује као судија са структуираним рубриком:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**Питања за размишљање:**
1. Када бисте више веровали проверaма базираним на правилима него LLM као судији?
2. Може ли модел поуздано оцењивати свој излаз? Која су ограничења?
3. Како се ово упоређује са шаблоном Editora из Парт 7?

### Вежба 4 - Упоредите варијанте упита

Пример покреће **две варијанте упита** за исте тест случајеве:

| Варијанта | Стил системског упита |
|-----------|---------------------|
| **Основна** | Генерички: „Ти си користан помоћник“ |
| **Специјализована** | Детаљна: „Ти си Zava DIY експерт који препоручује конкретне производе и пружа корак-по-корак упутства“ |

Након покретања, видећете табелу оцена као:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**Вежбе:**
1. Покрените процену и приметите оцене за сваку варијанту
2. Модификујте специјализовани упит да буде још прецизнији. Да ли оцена побољшава?
3. Додајте трећу варијанту упита и упоредите све три.
4. Покушајте да промените алијас модела (нпр. `phi-4-mini` против `phi-3.5-mini`) и упоредите резултате.

### Вежба 5 - Примените процену на свог агента

Користите оквир за процену као шаблон за ваше агенте:

1. **Дефинишите свој златни скуп података**: напишите 5 до 10 тест случајева са очекиваним кључним речима.
2. **Напишите системски упит**: инструкције за агента које желите да тестирате.
3. **Покрените процену**: добијте основне оцене.
4. **Умерено мењајте**: подесите упит, поново покрените и упоредите.
5. **Поставите праг**: нпр. „не пуштај у рад ако је укупна оцена испод 0.75“.

### Вежба 6 - Веза са Zava Editor шаблоном

Погледајте назад у Editor агента Zava Creative Writer-а (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# Уређивач је LLM као судија у продукцији:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

Ово је **исти концепт** као LLM као судија из Парт 8, али уграђен у производни ток, а не у офлајн скуп тестова. Обa образацa:

- Користе струкгуран JSON излаз из модела.
- Примењују критеријуме квалитета дефинисане у системском упиту.
- Доносе одлуку прођу/не прођу.

**Разлика:** Едитор ради у производњи (на сваки захтев). Оквир за процену ради у развоју (пре пуштања).

| **Процена пре пуштања** | Поставите прагове квалитета и контролишите издања на основу оцена |

## Следећи кораци

- **Проширите**: Додајте више тест случајева и категорија у златни скуп.
- **Аутоматизујте**: Интегришите процену у CI/CD процес.
- **Напредни судије**: Користите већи модел као судију код тестирања мањег модела.
- **Пратите током времена**: Сачувајте резултате процене за упоређивање варијанти упита и модела.
