# भाग 8: Foundry Local के साथ मूल्यांकन-नेतृत्व विकास

> **लक्ष्य:** एक मूल्यांकन फ्रेमवर्क बनाएं जो व्यवस्थित रूप से आपके AI एजेंट्स का परीक्षण और स्कोर करता है, एक ही लोकल मॉडल का उपयोग एजेंट के परीक्षण और जज दोनों के रूप में करता है, ताकि आप शिपिंग से पहले प्रांप्ट पर आत्मविश्वास के साथ पुनरावृत्ति कर सकें।

## मूल्यांकन-नेतृत्व विकास क्यों?

AI एजेंट बनाने में, "यह लगभग सही लगता है" पर्याप्त नहीं है। **मूल्यांकन-नेतृत्व विकास** एजेंट आउटपुट को कोड की तरह मानता है: आप पहले परीक्षण लिखते हैं, गुणवत्ता को मापते हैं, और तभी शिप करते हैं जब स्कोर थ्रेशोल्ड पूरा हो।

Zava Creative Writer (भाग 7) में, **एडिटर एजेंट** पहले से ही एक हल्का मूल्यांकनकर्ता के रूप में कार्य करता है; यह ACCEPT या REVISE का निर्णय देता है। यह लैब उस पैटर्न को एक पुनरावृत्त मूल्यांकन फ्रेमवर्क में औपचारिक बनाती है जिसे आप किसी भी एजेंट या पाइपलाइन पर लागू कर सकते हैं।

| समस्या | समाधान |
|---------|----------|
| प्रॉम्प्ट परिवर्तन चुपचाप गुणवत्ता को तोड़ते हैं | **गोल्डन डेटासेट** regresions पकड़ता है |
| "एक उदाहरण पर काम करता है" पूर्वाग्रह | **कई परीक्षण मामलों** किनारे के मामलों को उजागर करते हैं |
| व्यक्तिपरक गुणवत्ता आकलन | **नियम-आधारित + LLM-के-रूप में-जज स्कोरिंग** संख्याएं प्रदान करता है |
| प्रॉम्प्ट विकल्पों की तुलना करने का कोई तरीका नहीं | **साइड-बाय-साइड मूल्यांकन रन** समेकित स्कोर के साथ |

## प्रमुख अवधारणाएं

### 1. गोल्डन डेटासेट्स

एक **गोल्डन डेटासेट** परखा हुआ परीक्षण मामलों का चयनित सेट है जिसमें ज्ञात अपेक्षित आउटपुट होते हैं। प्रत्येक परीक्षण मामले में शामिल है:

- **इनपुट**: एजेंट को भेजा जाने वाला प्रॉम्प्ट या प्रश्न
- **अपेक्षित आउटपुट**: एक सही या उच्च-गुणवत्ता प्रतिक्रिया में क्या होना चाहिए (कीवर्ड, संरचना, तथ्य)
- **श्रेणी**: रिपोर्टिंग के लिए समूहबद्ध करना (जैसे "तथ्यात्मक सटीकता", "स्वर", "पूर्णता")

### 2. नियम-आधारित जांच

तेज, निर्धारक जांच जो LLM की आवश्यकता नहीं होती:

| जांच | यह क्या परीक्षण करता है |
|-------|--------------|
| **लंबाई की सीमाएं** | प्रतिक्रिया बहुत छोटी (आलसी) या बहुत लंबी (बहुत बकवास) नहीं है |
| **आवश्यक कीवर्ड** | प्रतिक्रिया अपेक्षित शब्दों या संस्थाओं का उल्लेख करती है |
| **फॉर्मेट सत्यापन** | JSON पार्स करने योग्य है, Markdown हेडर मौजूद हैं |
| **प्रतिबंधित सामग्री** | कोई कल्पित ब्रांड नाम नहीं, कोई प्रतियोगी उल्लेख नहीं |

### 3. LLM-के-रूप में-जज

अपने आउटपुट (या किसी दूसरे प्रॉम्प्ट विकल्प के आउटपुट) को ग्रेड करने के लिए **उसी लोकल मॉडल** का उपयोग करें। जज को मिलता है:

- मूल प्रश्न
- एजेंट की प्रतिक्रिया
- ग्रेडिंग मानदंड

और यह एक संरचित स्कोर लौटाता है। यह भाग 7 के एडिटर पैटर्न का आईना है लेकिन इसे परीक्षण सूट भर में व्यवस्थित रूप से लागू किया गया है।

### अभ्यास 2 - गोल्डन डेटासेट को समझें

मूल्यांकन नमूने में परिभाषित परीक्षण मामलों की जांच करें। प्रत्येक परीक्षण मामले में है:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**विचार करने के प्रश्न:**
1. अपेक्षित मान **कीवर्ड** क्यों हैं पूरे वाक्यों के बजाय?
2. विश्वसनीय मूल्यांकन के लिए आपको कितने परीक्षण मामलों की आवश्यकता है?
3. आप अपने स्वयं के अनुप्रयोग के लिए कौन से श्रेणियाँ जोड़ेंगे?

---

### अभ्यास 3 - नियम-आधारित बनाम LLM-के-रूप में-जज स्कोरिंग को समझें

मूल्यांकन फ्रेमवर्क **दो स्कोरिंग परतों** का उपयोग करता है:

#### नियम-आधारित जांचें (तेज, निर्धारक)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-के-रूप में-जज (सूक्ष्म, गुणात्मक)

उसी लोकल मॉडल का उपयोग रचनात्मक नियमिका के साथ एक जज के रूप में करता है:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**विचार करने के प्रश्न:**
1. आप कब नियम-आधारित जांचों पर LLM-के-रूप में-जज से अधिक भरोसा करेंगे?
2. क्या कोई मॉडल विश्वसनीय रूप से अपनी स्वयं की आउटपुट का मूल्यांकन कर सकता है? इसकी सीमाएँ क्या हैं?
3. यह भाग 7 के एडिटर एजेंट पैटर्न की तुलना में कैसा है?

---

### अभ्यास 4 - प्रॉम्प्ट विकल्पों की तुलना करें

नमूना **दो प्रॉम्प्ट विकल्पों** को उसी परीक्षण मामलों के खिलाफ चलाता है:

| विकल्प | सिस्टम प्रॉम्प्ट शैली |
|---------|-------------------|
| **बेसलाइन** | सामान्य: "आप एक सहायक सहायक हैं" |
| **विशेषीकृत** | विस्तृत: "आप एक Zava DIY विशेषज्ञ हैं जो विशिष्ट उत्पादों की सिफारिश करते हैं और चरण-दर-चरण मार्गदर्शन प्रदान करते हैं" |

चलाने के बाद, आप एक स्कोरकार्ड देखेंगे जैसा कि:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**अभ्यास:**
1. मूल्यांकन चलाएं और प्रत्येक विकल्प के लिए स्कोर नोट करें
2. विशेषीकृत प्रॉम्प्ट को और अधिक विशिष्ट बनाने के लिए संशोधित करें। क्या स्कोर बेहतर होता है?
3. तीसरा प्रॉम्प्ट विकल्प जोड़ें और सभी तीन की तुलना करें।
4. मॉडल उपनाम बदलने का प्रयास करें (जैसे `phi-4-mini` बनाम `phi-3.5-mini`) और परिणामों की तुलना करें।

### अभ्यास 5 - मूल्यांकन को अपने स्वयं के एजेंट पर लागू करें

मूल्यांकन फ्रेमवर्क का उपयोग अपने स्वयं के एजेंट्स के लिए एक टेम्पलेट के रूप में करें:

1. **अपना गोल्डन डेटासेट परिभाषित करें**: 5 से 10 परीक्षण मामले अपेक्षित कीवर्ड के साथ लिखें।
2. **अपना सिस्टम प्रॉम्प्ट लिखें**: उस एजेंट निर्देश जो आप परीक्षण करना चाहते हैं।
3. **मूल्यांकन चलाएं**: बेसलाइन स्कोर प्राप्त करें।
4. **पुनरावृत्ति करें**: प्रॉम्प्ट में संशोधन करें, पुनः चलाएं, और तुलना करें।
5. **एक थ्रेशोल्ड सेट करें**: जैसे "0.75 संयुक्त स्कोर से नीचे शिप न करें"।

### अभ्यास 6 - Zava एडिटर पैटर्न से कनेक्शन

Zava Creative Writer के एडिटर एजेंट (`zava-creative-writer-local/src/api/agents/editor/editor.py`) को देखें:

```python
# संपादक उत्पादन में न्यायाधीश के रूप में एक LLM है:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

यह भाग 8 के LLM-के-रूप में-जज जैसा ही **समान अवधारणा** है, लेकिन ऑफ़लाइन परीक्षण सूट में नहीं, बल्कि उत्पादन पाइपलाइन में निहित है। दोनों पैटर्न:

- मॉडल से संरचित JSON आउटपुट का उपयोग करते हैं।
- सिस्टम प्रॉम्प्ट में परिभाषित गुणवत्ता मानदंड लागू करते हैं।
- पास/फेल निर्णय लेते हैं।

**अंतर:** एडिटर उत्पादन में चलता है (हर अनुरोध पर)। मूल्यांकन फ्रेमवर्क विकास में चलता है (शिप करने से पहले)।

|---------|----------|
| **गोल्डन डेटासेट्स** | परीक्षण मामलों को पहले से चुनें; ये आपके AI के लिए रिग्रेशन टेस्ट हैं |
| **नियम-आधारित जांचें** | तेज, निर्धारक, और स्पष्ट त्रुटियों को पकड़ती हैं (लंबाई, कीवर्ड, फॉर्मेट) |
| **LLM-के-रूप में-जज** | उसी लोकल मॉडल का उपयोग करके सूक्ष्म गुणवत्ता स्कोरिंग |
| **प्रॉम्प्ट तुलना** | एक ही परीक्षण सूट के खिलाफ कई विकल्प चलाएं और सर्वश्रेष्ठ चुनें |
| **डिवाइस पर लाभ** | सभी मूल्यांकन स्थानीय रूप से चलता है: कोई API लागत नहीं, कोई दर सीमा नहीं, कोई डेटा मशीन से बाहर नहीं जाता |
| **शिप से पहले मूल्यांकन** | गुणवत्ता थ्रेशोल्ड सेट करें और मूल्यांकन स्कोर पर रिलीज़ रोकें |

## अगले कदम

- **स्केल अप करें**: अपने गोल्डन डेटासेट में अधिक परीक्षण मामले और श्रेणियाँ जोड़ें।
- **स्वचालित करें**: मूल्यांकन को अपने CI/CD पाइपलाइन में एकीकृत करें।
- **उन्नत जज**: एक बड़े मॉडल का उपयोग करें जज के रूप में जबकि एक छोटे मॉडल के आउटपुट का परीक्षण करते हैं।
- **समय के साथ ट्रैक करें**: मूल्यांकन परिणाम सेव करें ताकि प्रॉम्प्ट और मॉडल संस्करणों के बीच तुलना की जा सके।
