# ส่วนที่ 8: การพัฒนาที่ขับเคลื่อนด้วยการประเมินผลกับ Foundry Local

> **เป้าหมาย:** สร้างกรอบการประเมินที่ทดสอบและให้คะแนนตัวแทน AI ของคุณอย่างเป็นระบบ โดยใช้โมเดลท้องถิ่นเดียวกันทั้งตัวแทนที่ถูกทดสอบและผู้ตัดสิน เพื่อให้คุณสามารถปรับแต่งพรอมต์ได้อย่างมั่นใจก่อนปล่อยใช้งาน

## ทำไมต้องการพัฒนาที่ขับเคลื่อนด้วยการประเมินผล?

เมื่อสร้างตัวแทน AI นั้น "ดูเหมือนจะถูกต้อง" ไม่เพียงพอ **การพัฒนาที่ขับเคลื่อนด้วยการประเมินผล** จะถือว่าผลลัพธ์ของตัวแทนเหมือนกับโค้ด: คุณเขียนการทดสอบก่อน วัดคุณภาพ และส่งมอบเมื่อคะแนนถึงเกณฑ์ที่กำหนด

ใน Zava Creative Writer (ส่วนที่ 7) ตัวแทน **Editor** ทำหน้าที่เป็นผู้ประเมินน้ำหนักเบาอยู่แล้ว; มันตัดสินใจว่า ACCEPT หรือ REVISE ห้องปฏิบัติการนี้ทำให้รูปแบบนั้นเป็นกรอบการประเมินที่ทำซ้ำได้ซึ่งคุณสามารถประยุกต์ใช้กับตัวแทนหรือสายงานใดก็ได้

| ปัญหา | วิธีแก้ไข |
|---------|----------|
| การเปลี่ยนพรอมต์ทำให้คุณภาพลดลงโดยไม่รู้ตัว | **ชุดข้อมูลทองคำ** จับความล่าช้าได้ |
| อคติ "ทำงานได้แค่ตัวอย่างเดียว" | **หลายกรณีทดสอบ** รายงานกรณีขอบเขต |
| การประเมินคุณภาพแบบอัตวิสัย | **กฎเกณฑ์ + การให้คะแนนโดย LLM ในฐานะผู้ตัดสิน** ให้ตัวเลขชัดเจน |
| ไม่มีวิธีเปรียบเทียบตัวแปรพรอมต์ | **รันประเมินผลข้างกัน** พร้อมคะแนนรวม |

## แนวคิดสำคัญ

### 1. ชุดข้อมูลทองคำ

**ชุดข้อมูลทองคำ** คือชุดกรณีทดสอบที่คัดสรรมาโดยมีผลลัพธ์ที่คาดหวังทราบแล้ว กรณีทดสอบแต่ละรายการประกอบด้วย:

- **อินพุต:** คำสั่งหรือคำถามที่จะส่งให้ตัวแทน
- **ผลลัพธ์ที่คาดหวัง:** สิ่งที่คำตอบที่ถูกต้องหรือคุณภาพสูงควรมี (คำสำคัญ โครงสร้าง ข้อเท็จจริง)
- **หมวดหมู่:** การจัดกลุ่มสำหรับรายงาน (เช่น "ความถูกต้องของข้อเท็จจริง" "โทนเสียง" "ความสมบูรณ์")

### 2. การตรวจด้วยกฎเกณฑ์

การตรวจสอบที่รวดเร็วและกำหนดผลชัดเจนโดยไม่ต้องใช้ LLM:

| การตรวจ | สิ่งที่ทดสอบ |
|-------|--------------|
| **ขอบเขตความยาว** | คำตอบไม่สั้นเกินไป (ขาดความพยายาม) หรือยาวเกินไป (เยิ่นเย้อ) |
| **คำสำคัญที่จำเป็น** | คำตอบกล่าวถึงคำหรือเอนทิตี้ที่คาดหวัง |
| **การตรวจสอบรูปแบบ** | JSON สามารถแปลงได้ หัวข้อ Markdown มีอยู่ |
| **เนื้อหาที่ถูกห้าม** | ไม่มีแบรนด์ที่จินตนาการขึ้น ไม่มีการกล่าวถึงคู่แข่ง |

### 3. LLM ในฐานะผู้ตัดสิน

ใช้ **โมเดลท้องถิ่นเดียวกัน** เพื่อให้คะแนนผลลัพธ์ของตัวเอง (หรือผลลัพธ์จากพรอมต์ตัวแปรอื่น) ผู้ตัดสินจะได้รับ:

- คำถามต้นฉบับ
- คำตอบของตัวแทน
- เกณฑ์การให้คะแนน

และคืนคะแนนในโครงสร้าง นี่สะท้อนรูปแบบของ Editor ในส่วนที่ 7 แต่ถูกประยุกต์อย่างเป็นระบบทั่วทั้งชุดทดสอบ

### แบบฝึกหัด 1 - รันกรอบการประเมินผล

### แบบฝึกหัด 2 - ทำความเข้าใจชุดข้อมูลทองคำ

ตรวจสอบกรณีทดสอบที่กำหนดในตัวอย่างการประเมิน ผลลัพธ์แต่ละกรณีมี:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**คำถามสำหรับพิจารณา:**
1. ทำไมค่าที่คาดหวังจึงเป็น **คำสำคัญ** แทนประโยคเต็ม?
2. ต้องใช้กี่กรณีทดสอบจึงจะประเมินได้อย่างน่าเชื่อถือ?
3. คุณจะเพิ่มหมวดหมู่ใดสำหรับแอปพลิเคชันของคุณเอง?

---

### แบบฝึกหัด 3 - เข้าใจการให้คะแนนตามกฎเกณฑ์และ LLM ในฐานะผู้ตัดสิน

กรอบการประเมินใช้ **สองชั้นการให้คะแนน**:

#### การตรวจสอบตามกฎเกณฑ์ (รวดเร็ว กำหนดผลชัดเจน)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM ในฐานะผู้ตัดสิน (ลึกซึ้ง คุณภาพ)

โมเดลท้องถิ่นเดียวกันทำหน้าที่เป็นผู้ตัดสินโดยใช้เครื่องมือวัดโครงสร้าง:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**คำถามสำหรับพิจารณา:**
1. คุณจะเชื่อการตรวจสอบตามกฎเกณฑ์มากกว่า LLM ในสถานการณ์ใด?
2. โมเดลจะตัดสินผลลัพธ์ของตัวเองได้อย่างเชื่อถือได้หรือไม่? ข้อจำกัดมีอะไรบ้าง?
3. สิ่งนี้เปรียบเทียบกับรูปแบบ Editor จากส่วนที่ 7 อย่างไร?

---

### แบบฝึกหัด 4 - เปรียบเทียบตัวแปรพรอมต์

ตัวอย่างรัน **สองตัวแปรพรอมต์** กับกรณีทดสอบเดียวกัน:

| ตัวแปร | สไตล์พรอมต์ระบบ |
|---------|-------------------|
| **ฐาน** | ทั่วไป: "คุณคือผู้ช่วยที่เป็นประโยชน์" |
| **เฉพาะทาง** | ละเอียด: "คุณคือผู้เชี่ยวชาญ Zava DIY ที่แนะนำผลิตภัณฑ์เฉพาะและให้คำแนะนำทีละขั้นตอน" |

หลังรันจะเห็นการ์ดคะแนนแบบนี้:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**แบบฝึกหัด:**
1. รันการประเมินและบันทึกคะแนนของแต่ละตัวแปร
2. แก้ไขพรอมต์เฉพาะทางให้เจาะจงมากขึ้น คะแนนดีขึ้นหรือไม่?
3. เพิ่มตัวแปรพรอมต์ที่สามและเปรียบเทียบทั้งสามตัว
4. ลองเปลี่ยนชื่อโมเดล (เช่น `phi-4-mini` กับ `phi-3.5-mini`) แล้วเปรียบเทียบผลลัพธ์

### แบบฝึกหัด 5 - ใช้การประเมินกับตัวแทนของคุณเอง

ใช้กรอบการประเมินเป็นต้นแบบสำหรับตัวแทนของคุณเอง:

1. **กำหนดชุดข้อมูลทองคำของคุณ:** เขียน 5 ถึง 10 กรณีทดสอบพร้อมคำสำคัญที่คาดหวัง
2. **เขียนพรอมต์ระบบของคุณ:** คำสั่งตัวแทนที่ต้องการทดสอบ
3. **รันการประเมิน:** รับคะแนนฐาน
4. **ปรับปรุง:** ปรับพรอมต์ รันซ้ำ และเปรียบเทียบ
5. **ตั้งเกณฑ์:** เช่น "ไม่ส่งออกถ้าคะแนนรวมต่ำกว่า 0.75"

### แบบฝึกหัด 6 - การเชื่อมโยงกับรูปแบบ Editor ของ Zava

ย้อนกลับไปดู Editor agent ของ Zava Creative Writer (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# ตัวแก้ไขเป็น LLM-แบบผู้ตัดสินในระบบการผลิต:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

นี่เป็น **แนวคิดเดียวกัน** กับ LLM ในฐานะผู้ตัดสินของส่วนที่ 8 แต่ฝังอยู่ในกระบวนการผลิตแทนที่จะเป็นชุดทดสอบออฟไลน์ ทั้งสองรูปแบบ:

- ใช้ผลลัพธ์ JSON ที่มีโครงสร้างจากโมเดล
- ใช้เกณฑ์คุณภาพที่กำหนดในพรอมต์ระบบ
- ตัดสินผ่าน/ไม่ผ่าน

**ความแตกต่าง:** Editor รันในผลิตจริง (ทุกคำขอ) กรอบการประเมินรันในการพัฒนา (ก่อนปล่อย)

## ข้อสรุปสำคัญ

| แนวคิด | ข้อสรุป |
|---------|----------|
| **ชุดข้อมูลทองคำ** | คัดกรองกรณีทดสอบตั้งแต่ต้น พวกเขาคือการทดสอบความล่าช้า AI ของคุณ |
| **การตรวจสอบด้วยกฎเกณฑ์** | รวดเร็ว กำหนดผลชัด และจับความผิดพลาดที่ชัดเจน (ความยาว คำสำคัญ รูปแบบ) |
| **LLM ในฐานะผู้ตัดสิน** | ให้คะแนนคุณภาพอย่างละเอียดโดยใช้โมเดลท้องถิ่นเดียวกัน |
| **การเปรียบเทียบพรอมต์** | รันตัวแปรหลายชุดกับชุดทดสอบเดียวกันเพื่อเลือกดีที่สุด |
| **ข้อได้เปรียบบนอุปกรณ์** | การประเมินทั้งหมดทำบนเครื่องท้องถิ่น: ไม่มีค่าใช้จ่าย API, ไม่มีข้อจำกัดอัตรา, ไม่มีข้อมูลออกนอกเครื่อง |
| **ประเมินก่อนปล่อย** | ตั้งเกณฑ์คุณภาพและควบคุมการปล่อยบนคะแนนประเมิน |

## ขั้นตอนต่อไป

- **ขยายขนาด:** เพิ่มกรณีทดสอบและหมวดหมู่ในชุดข้อมูลทองคำของคุณ
- **ทำอัตโนมัติ:** ผนวกการประเมินเข้ากับกระบวนการ CI/CD ของคุณ
- **ผู้ตัดสินขั้นสูง:** ใช้โมเดลใหญ่กว่าในฐานะผู้ตัดสินขณะทดสอบผลลัพธ์โมเดลเล็กกว่า
- **ติดตามผลตลอดเวลา:** บันทึกผลประเมินเพื่อเปรียบเทียบพรอมต์และเวอร์ชันโมเดล
