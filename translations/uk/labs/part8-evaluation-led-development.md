# Частина 8: Розробка, керована оцінкою, з Foundry Local

> **Мета:** Створити фреймворк оцінки, який систематично тестує і оцінює ваших AI агентів, використовуючи той самий локальний модель як агента під тестуванням, так і суддю, щоб ви могли впевнено ітерувати запити перед випуском.

## Чому розробка, керована оцінкою?

При створенні AI агентів формулювання «виглядає приблизно правильно» недостатньо. **Розробка, керована оцінкою**, ставиться до вихідних даних агента як до коду: спочатку пишете тести, вимірюєте якість і відпускаєте тільки, коли оцінки досягають порогу.

У Zava Creative Writer (Частина 7) **агент Редактор** вже діє як легкий оцінювач; він вирішує, приймати відповідь чи редагувати її. Ця лабораторія формалізує цей патерн у багаторазовий фреймворк оцінки, який можна застосувати до будь-якого агента чи pipeline.

| Проблема | Рішення |
|---------|----------|
| Зміни в запиті непомітно руйнують якість | **Золотий датасет** ловить регресії |
| Упередженість «працює на одному прикладі» | **Кілька тест-кейсів** виявляють крайні випадки |
| Суб’єктивна оцінка якості | **Оцінка на основі правил + LLM як суддя** дають цифри |
| Відсутність способу порівняти варіанти запитів | **Паралельні запуски оцінки** з підсумковими результатами |

---

## Ключові поняття

### 1. Золоті датасети

**Золотий датасет** — це відібраний набір тест-кейсів із відомими очікуваними результатами. Кожен кейс містить:

- **Вхідні дані**: запит або питання, яке відправляється агенту
- **Очікуваний результат**: те, що правильна або висока якість відповіді повинна містити (ключові слова, структуру, факти)
- **Категорія**: групування для звітності (наприклад, «фактична точність», «тон», «повнота»)

### 2. Перевірки на основі правил

Швидкі, детерміністичні перевірки, які не потребують LLM:

| Перевірка | Що тестує |
|-------|--------------|
| **Обмеження довжини** | Відповідь не занадто коротка (лінива) або занадто довга (заплутана) |
| **Обов’язкові ключові слова** | Відповідь містить очікувані терміни або сутності |
| **Валідація формату** | JSON розбирається, присутні заголовки Markdown |
| **Заборонений вміст** | Без вигаданих брендів, без згадок конкурентів |

### 3. LLM як суддя

Використовуйте **той самий локальний модель** для оцінювання власних результатів (або результатів іншого варіанту запиту). Суддя отримує:

- Початкове питання
- Відповідь агента
- Критерії оцінювання

І повертає структуровану оцінку. Це повторює патерн Редактора з Частини 7, але систематично застосовується по всьому тестовому набору.

### Вправа 2 – Розуміння золотого датасету

Розгляньте тест-кейси, визначені у прикладі оцінки. Кожен кейс має:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**Питання для роздумів:**
1. Чому очікувані значення — це **ключові слова**, а не повні речення?
2. Скільки тест-кейсів потрібно для надійної оцінки?
3. Які категорії ви б додали для власного застосування?

---

### Вправа 3 – Розуміння оцінки на основі правил і LLM як судді

Фреймворк оцінки використовує **два рівні оцінювання**:

#### Перевірки на основі правил (швидкі, детерміністичні)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM як суддя (більш точне, якісне)

Той самий локальний модель виступає суддею із структурованою рубрикою:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**Питання для роздумів:**
1. Коли ви б довіряли перевіркам на основі правил більше, ніж LLM як судді?
2. Чи може модель надійно оцінювати власні результати? Які обмеження?
3. Як це порівнюється з патерном Редактора з Частини 7?

### Вправа 4 – Порівняння варіантів запиту

Приклад запускає **два варіанти запиту** для цих же тест-кейсів:

| Варіант | Стиль системного запиту |
|---------|-------------------|
| **Базовий** | Загальний: «Ви корисний помічник» |
| **Спеціалізований** | Детальний: «Ви експерт Zava DIY, який рекомендує конкретні продукти і дає покрокові інструкції» |

Після запуску ви побачите таблицю на кшталт:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**Завдання:**
1. Запустіть оцінку і зафіксуйте оцінки для кожного варіанту
2. Змініть спеціалізований запит, зробивши його ще детальнішим. Чи покращилась оцінка?
3. Додайте третій варіант запиту і порівняйте всі три.
4. Спробуйте змінити псевдонім моделі (наприклад, `phi-4-mini` проти `phi-3.5-mini`) і порівняйте результати.

### Вправа 5 – Застосування оцінки до власного агента

Використайте фреймворк оцінки як шаблон для своїх агентів:

1. **Визначте свій золотий датасет**: напишіть 5–10 тест-кейсів з очікуваними ключовими словами.
2. **Напишіть системний запит**: інструкції для агента, які хочете протестувати.
3. **Запустіть оцінку**: отримаєте базові оцінки.
4. **Ітеруйте**: відкоригуйте запит, знову запустіть і порівняйте.
5. **Встановіть поріг**: наприклад, «не випускати, якщо сумарна оцінка менша за 0.75».

---

### Вправа 6 – Зв’язок з патерном редактора Zava

Погляньте назад на агента Редактора у Zava Creative Writer (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# Редактор є LLM як суддя у виробництві:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

Це **той самий концепт**, що і LLM як суддя у Частині 8, але вбудований у продакшн-процес, а не в офлайн тестовий набір. Обидва патерни:

- Використовують структурований JSON-вихід від моделі.
- Застосовують критерії якості, визначені у системному запиті.
- Приймають рішення про успішність або провал.

**Відмінність:** Редактор працює в продакшні (на кожному запиті). Фреймворк оцінки — у розробці (до релізу).

## Головні висновки

| Поняття | Висновок |
|---------|----------|
| **Золоті датасети** | Раннє формування тестів; це ваші регресійні тести для AI |
| **Перевірки на основі правил** | Швидкі, детерміністичні, ловлять очевидні збої (довжина, ключові слова, формат) |
| **LLM як суддя** | Тонке оцінювання якості тим самим локальним моделью |
| **Порівняння запитів** | Запускайте кілька варіантів на одному наборі, щоб обрати найкращий |
| **Перевага локального запуска** | Уся оцінка виконується локально: нема витрат на API, нема лімітів, дані не покидають машину |
| **Оцінка перед випуском** | Встановіть пороги якості й блоковуйте релізи, якщо оцінки нижчі за них |

## Наступні кроки

- **Масштабуйте**: додайте більше тест-кейсів і категорій у золотий датасет.
- **Автоматизуйте**: інтегруйте оцінку у ваш CI/CD pipeline.
- **Просунуті судді**: використовуйте більшу модель як суддю, тестуючи вивід меншої моделі.
- **Відстежуйте зміни**: зберігайте результати оцінки для порівняння версій запитів і моделей.
