# حصہ 8: Foundry Local کے ساتھ جائزہ پر مبنی ترقی

> **مقصد:** ایک جائزہ فریم ورک تیار کریں جو آپ کے AI ایجنٹس کا منظم انداز میں ٹیسٹ اور اسکور کرے، اسی لوکل ماڈل کو بطور ایجنٹ اور جج دونوں استعمال کرتے ہوئے، تاکہ آپ پرامپٹس پر اعتماد کے ساتھ دوبارہ کام کر سکیں اور شپ کرنے سے قبل انہیں بہتر بنا سکیں۔

## جائزہ پر مبنی ترقی کیوں؟

جب AI ایجنٹس تیار کر رہے ہوں تو "یہ تقریباً ٹھیک لگ رہا ہے" کافی نہیں ہے۔ **جائزہ پر مبنی ترقی** ایجنٹ کے آؤٹ پٹس کو کوڈ کی طرح دیکھتی ہے: آپ پہلے ٹیسٹ لکھتے ہیں، معیار کی پیمائش کرتے ہیں، اور صرف تبھی بھیجتے ہیں جب اسکور ایک حد کو پورا کریں۔

Zava Creative Writer (حصہ 7) میں، **ایڈیٹر ایجنٹ** پہلے سے ہی ایک ہلکا پھلکا جائزہ لینے والا کے طور پر کام کرتا ہے؛ یہ ACCEPT یا REVISE کا فیصلہ کرتا ہے۔ یہ لیب اس پیٹرن کو رسمی شکل دیتی ہے ایک قابل دہرائش جائزہ فریم ورک میں جسے آپ کسی بھی ایجنٹ یا پائپ لائن پر لاگو کر سکتے ہیں۔

| مسئلہ | حل |
|---------|----------|
| پرامپٹ میں تبدیلیاں خاموشی سے معیار کو توڑتی ہیں | **گولڈن ڈیٹاسیٹ** ریگریشن پکڑتی ہے |
| "ایک مثال پر کام کرنا" کی تعصب | **متعدد ٹیسٹ کیسز** کنارے کے معاملات ظاہر کرتے ہیں |
| ذاتی معیار کی تشخیص | **قاعدہ پر مبنی + LLM-بحیثیت جج اسکورنگ** اعداد و شمار فراہم کرتا ہے |
| پرامپٹ متغیرات کا کوئی موازنہ کا طریقہ نہیں | **ساتھ ساتھ جائزہ چلانا** مجموعی اسکور کے ساتھ |

## کلیدی تصورات

### 1. گولڈن ڈیٹاسیٹس

ایک **گولڈن ڈیٹاسیٹ** ٹیسٹ کیسز کا ایک مرتب شدہ مجموعہ ہے جس کے متوقع نتائج معلوم ہوں۔ ہر ٹیسٹ کیس میں شامل ہیں:

- **ان پٹ**: وہ پرامپٹ یا سوال جو ایجنٹ کو بھیجا جائے
- **متوقع آؤٹ پٹ**: وہ صحیح یا اعلی معیار کا جواب جو ہونا چاہیے (کلیدی الفاظ، ساخت، حقائق)
- **زمرہ**: رپورٹنگ کے لیے گروپ بندی (مثلاً "حقیقی درستگی"، "لہجہ"، "مکمل پن")

### 2. قاعدہ پر مبنی چیکس

تیز، جزمیت پسند چیکس جو LLM کی ضرورت نہیں رکھتے:

| چیک | یہ کیا ٹیسٹ کرتا ہے |
|-------|--------------|
| **لمبائی کی حد** | جواب نہ تو بہت چھوٹا (سست) اور نہ ہی بہت لمبا (طویل بیانیہ) ہو |
| **ضروری کلیدی الفاظ** | جواب میں متوقع اصطلاحات یا موجودات کا ذکر ہو |
| **فارمیٹ کی تصدیق** | JSON قابل پارس ہونا چاہیے، Markdown ہیڈرز موجود ہوں |
| **ممنوعہ مواد** | کوئی مصنوعی برانڈ نام، یا مقابل کمپنیاں ذکر نہیں |

### 3. LLM-بحیثیت جج

اسی **لوکل ماڈل** کا استعمال کرکے اس کے اپنے آؤٹ پٹس (یا کسی مختلف پرامپٹ ورینٹ کے آؤٹ پٹس) کو گریڈ کریں۔ جج کو موصول ہوتا ہے:

- اصل سوال
- ایجنٹ کا جواب
- گریڈنگ کے معیار

اور یہ ایک منظم اسکور واپس کرتا ہے۔ یہ حصہ 7 کے ایڈیٹر پیٹرن کی عکاسی کرتا ہے مگر مناسب طریقے سے پورے ٹیسٹ سوئٹ پر لاگو ہوتا ہے۔

### مشق 2 - گولڈن ڈیٹاسیٹ کو سمجھیں

جائزہ نمونے میں تعریف شدہ ٹیسٹ کیسز کا معائنہ کریں۔ ہر ٹیسٹ کیس میں شامل ہیں:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**سوچنے کے سوالات:**
1. متوقع اقدار **کلیدی الفاظ** کیوں ہیں بجائے مکمل جملوں کے؟
2. قابل اعتماد جائزے کے لیے آپ کو کتنے ٹیسٹ کیسز کی ضرورت ہے؟
3. آپ اپنی ایپلیکیشن کے لیے کون سے زمرے شامل کریں گے؟

### مشق 3 - قاعدہ پر مبنی اور LLM-بحیثیت جج اسکورنگ کو سمجھیں

جائزہ فریم ورک میں **دو اسکورنگ پرتیں** استعمال ہوتی ہیں:

#### قاعدہ پر مبنی چیکس (تیز، جزمیت پسند)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-بحیثیت جج (باریک بینی سے، معیاری)

اسی لوکل ماڈل کو جج کی طرح منظم روبریک کے ساتھ استعمال کیا جاتا ہے:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**سوچنے کے سوالات:**
1. آپ کب قاعدہ پر مبنی چیکس پر LLM-بحیثیت جج سے زیادہ اعتماد کریں گے؟
2. کیا ایک ماڈل اپنے ہی آؤٹ پٹ کو قابل اعتماد طریقے سے جانچ سکتا ہے؟ اس کی حدود کیا ہیں؟
3. یہ حصّہ 7 کے ایڈیٹر ایجنٹ پیٹرن سے کیسے موازنہ کرتا ہے؟

### مشق 4 - پرامپٹ ورینٹس کا موازنہ کریں

نمونہ ایک ہی ٹیسٹ کیسز کے خلاف **دو پرامپٹ ورینٹس** چلاتا ہے:

| ورینٹ | سسٹم پرامپٹ کا انداز |
|---------|-------------------|
| **بیس لائن** | عمومی: "آپ ایک مددگار اسسٹنٹ ہیں" |
| **ماہرانہ** | تفصیلی: "آپ ایک Zava DIY ماہر ہیں جو مخصوص مصنوعات کی سفارش اور مرحلہ وار رہنمائی فراہم کرتے ہیں" |

چلانے کے بعد، آپ کو اسکور کارڈ جیسا جائے گا:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**مشقیں:**
1. جائزہ چلائیں اور ہر ورینٹ کے اسکور نوٹ کریں
2. ماہرانہ پرامپٹ کو مزید مخصوص کریں۔ کیا اسکور بہتر ہوتا ہے؟
3. تیسرا پرامپٹ ورینٹ شامل کریں اور تینوں کا موازنہ کریں۔
4. ماڈل عرف نام تبدیل کر کے دیکھیں (مثلاً `phi-4-mini` بمقابلہ `phi-3.5-mini`) اور نتائج کا موازنہ کریں۔

### مشق 5 - جائزہ کو اپنے ایجنٹ پر لاگو کریں

اپنے ایجنٹس کے لیے جائزہ فریم ورک کو ٹیمپلیٹ کی طرح استعمال کریں:

1. **اپنا گولڈن ڈیٹاسیٹ وضع کریں**: 5 سے 10 ٹیسٹ کیسز کلیدی الفاظ کے ساتھ لکھیں۔
2. **اپنا سسٹم پرامپٹ لکھیں**: وہ ہدایات جو آپ ایجنٹ کو دینا چاہتے ہیں۔
3. **جائزہ چلائیں**: بیس لائن اسکور حاصل کریں۔
4. **دوبارہ کام کریں**: پرامپٹ میں تبدیلی کریں، دوبارہ چلائیں، اور موازنہ کریں۔
5. **حد مقرر کریں**: مثلاً "0.75 کے مجموعی اسکور سے نیچے نہ بھیجیں"۔

### مشق 6 - Zava ایڈیٹر پیٹرن سے تعلق

Zava Creative Writer کا ایڈیٹر ایجنٹ دیکھیں (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# ایڈیٹر پروڈکشن میں ایک LLM بطور جج ہے:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

یہ وہ **ہی تصور** ہے جیسا کہ حصہ 8 کا LLM-بحیثیت جج، مگر یہ پیداوار پائپ لائن میں سرکاری طور پر استعمال ہوتا ہے بجائے آف لائن ٹیسٹ سوئٹ کے۔ دونوں پیٹرنز:

- ماڈل سے منظم JSON آؤٹ پٹ استعمال کرتے ہیں۔
- معیار کے معیارات سسٹم پرامپٹ میں متعین کرتے ہیں۔
- پاس/فیل فیصلہ کرتے ہیں۔

**فرق:** ایڈیٹر ہر درخواست پر پیداوار میں چلتا ہے۔ جائزہ فریم ورک ترقی میں چلتا ہے (شپ کرنے سے پہلے)۔

## اہم نکات

| تصور | نتیجہ |
|---------|----------|
| **گولڈن ڈیٹاسیٹس** | ٹیسٹ کیسز کو ابتدائی طور پر مرتب کریں؛ یہ آپ کے AI کے لیے ریگریشن ٹیسٹ ہیں |
| **قاعدہ پر مبنی چیکس** | تیز، یقینی، اور واضح ناکامیوں کو پکڑتے ہیں (لمبائی، کلیدی الفاظ، فارمیٹ) |
| **LLM-بحیثیت جج** | باریکی سے معیار کے اسکور اسی لوکل ماڈل کا استعمال کرتے ہوئے دیتے ہیں |
| **پرامپٹ موازنہ** | متعدد ورینٹس کو ایک ہی ٹیسٹ سوئٹ کے خلاف چلائیں تاکہ بہترین کو منتخب کریں |
| **ڈیوائس پر فائدہ** | تمام جائزے مقامی طور پر چلتے ہیں: کوئی API قیمت نہیں، کوئی ریٹ حد نہیں، کوئی ڈیٹا آپ کی مشین سے باہر نہیں جاتا |
| **جائزہ-قبل-شپ** | معیار کے حد مقرر کریں اور جائزہ اسکور پر ریلیزز محدود کریں |
