# भाग 8: Foundry Local सह मूल्यांकन-आधारित विकास

> **उद्दिष्ट:** आपल्या AI एजंट्सची सिस्टमॅटिक चाचणी आणि स्कोअर करण्यासाठी एक मूल्यांकन फ्रेमवर्क तयार करा, ज्यामध्ये एजंट आणि न्यायाधीश म्हणून एकसारख्या स्थानिक मॉडेलचा वापर केला जातो, त्यामुळे आपण खात्रीने प्रॉम्प्ट्सवर पुनरावृत्ती करू शकता आणि नंतर वितरित करू शकता.

## मूल्यांकन-आधारित विकास का?

AI एजंट्स तयार करताना, "हे बरोबर दिसते" हे पुरेसे नाही. **मूल्यांकन-आधारित विकास** एजंटच्या आउटपुट्सना कोडप्रमाणे घेतो: आपण प्रथम चाचण्या लिहिता, गुणवत्ता मोजता आणि स्कोअर विशिष्ट मानक गाठल्यावरच वितरित करता.

Zava Creative Writer (भाग 7) मध्ये, **संपादक एजंट** आधीच एक हलक्या प्रमाणावर मूल्यांकनकर्ता म्हणून काम करतो; तो ACCEPT किंवा REVISE याचा निर्णय घेतो. हा प्रयोग त्या पद्धतीला कोणत्याही एजंट किंवा पाइपलाइनवर लागू करता येण्याजोग्या पुनरावृत्त मूल्यांकन फ्रेमवर्कमध्ये रूपांतरित करतो.

| समस्या | उपाय |
|---------|----------|
| प्रॉम्प्ट बदलांमुळे गुपचूप गुणवत्ता खराब होणे | **Golden dataset** रिग्रेशन पकडतो |
| "एका उदाहरणावर काम करते" अशी बाजू | **एकाधिक चाचणी प्रकरणे** कडेकोर प्रकरणे उघडतात |
| व्यक्तिपरक गुणवत्ता मुल्यमापन | **नियमाधारित + LLM-जज स्कोअरिंग** संख्यात्मक पुरवठा करते |
| प्रॉम्प्ट वेरिएंटची तुलना करण्याचा मार्ग नाही | **साइड-बाय-साइड मूल्यांकन चालवणे** एकत्रित स्कोअरसह |

## मुख्य संकल्पना

### 1. Golden Datasets

**Golden dataset** म्हणजे पूर्वनिर्धारित अपेक्षित आउटपुट असलेल्या चाचणी प्रकरणांचा संग्रह. प्रत्येक चाचणी प्रकरणात:

- **इनपुट**: एजंटला पाठवायचा प्रॉम्प्ट किंवा प्रश्न
- **अपेक्षित आउटपुट**: योग्य किंवा उच्च-गुणवत्तेचा प्रतिसाद काय असावा (कीवर्ड्स, रचना, तथ्य)
- **श्रेणी**: अहवालासाठी गट (उदा. "तथ्यात्मक अचूकता", "स्वर", "पूर्णता")

### 2. नियमाधारित तपासण्या

वेगाने, निश्चितपणे तपासण्या ज्यांना LLM ची गरज नाही:

| तपासणी | काय तपासते |
|-------|--------------|
| **लांबीची मर्यादा** | प्रतिसाद फार लहान (सोडीवट) किंवा फार लांब (फालतू) नाही |
| **आवश्यक कीवर्ड्स** | प्रतिसादात अपेक्षित शब्द किंवा घटक असले पाहिजेत |
| **फॉरमॅट तपासणी** | JSON पार्स करणे शक्य, Markdown हेडर उपस्थित आहेत |
| **आकर्षक नव्हतेले विषय** | कोणतेही भासवलेले ब्रँड नाव नाही, स्पर्धक उल्लेख नाही |

### 3. LLM-जज

त्याच स्थानिक मॉडेलचा वापर करून स्वत: च्या किंवा वेगळ्या प्रॉम्प्ट वेरिएंटचे आउटपुट ग्रेड करा. न्यायाधीश यांना मिळते:

- मूळ प्रश्न
- एजंटचा प्रतिसाद
- ग्रेडिंग निकष

आणि तो एक संरचित स्कोअर परत करतो. हा भाग 7 मधील संपादक पद्धतीसारखा आहे परंतु संपूर्ण चाचणी संचावर लागू केलेले.

### प्रयोग 2 - Golden Dataset समजून घ्या

मूल्यांकन नमुन्यामध्ये निश्चित केलेली चाचणी प्रकरणे तपासा. प्रत्येकात आहे:

```
{
  "input":    "What tools do I need to build a deck?",
  "expected": ["saw", "drill", "screws", "level"],
  "category": "product-recommendation"
}
```

**विचार करण्यासाठी प्रश्न:**
1. अपेक्षित मूल्ये पूर्ण वाक्यांऐवजी का **कीवर्ड्स** आहेत?
2. विश्वासार्ह मूल्यांकनासाठी किती चाचणी प्रकरणे आवश्यक आहेत?
3. आपण आपल्या अनुप्रयोगासाठी कोणत्या श्रेण्या जोडाल?

---

### प्रयोग 3 - नियमाधारित व LLM-जज स्कोअरिंग समजून घ्या

मूल्यांकन फ्रेमवर्कमध्ये **दोन स्कोअरिंग स्तर** वापरले आहेत:

#### नियमाधारित तपासण्या (वेगवान, निश्चित)

```
✓ Length: 50-500 words       → 1 point
✓ Keywords: 3/4 found        → 0.75 points  
✗ Forbidden: mentions "Home Depot" → 0 points
─────────────────────────────
Rule score: 0.58 / 1.0
```

#### LLM-जज (सूक्ष्म, गुणात्मक)

त्याच स्थानिक मॉडेलला रचनात्मक निकषांसह न्यायाधीश म्हणून वापरा:

```
Rate this response on a scale of 1-5:
- 1: Completely wrong or irrelevant
- 2: Partially correct but missing key information
- 3: Adequate but could be improved
- 4: Good response with minor issues
- 5: Excellent, comprehensive, well-structured

Score: 4
Reasoning: The response correctly identifies all necessary tools
and provides practical advice, but could include safety equipment.
```

**विचार करण्यासाठी प्रश्न:**
1. नियमाधारित तपासण्यांवर कधी LLM-जज वर विश्वास ठेवाल?
2. मॉडेल स्वतःच्या आउटपुटचे विश्वसनीयपणे मूल्यांकन करू शकते का? त्याची मर्यादा काय आहे?
3. भाग 7 मधील संपादक एजंट पद्धतीशी याची कशी तुलना करता येईल?

---

### प्रयोग 4 - प्रॉम्प्ट वेरिएंट्सची तुलना करा

नमुना **दोन प्रॉम्प्ट वेरिएंट्स** त्याच चाचणी प्रकरणांवर चालवतो:

| वेरिएंट | सिस्टम प्रॉम्प्ट शैली |
|---------|-------------------|
| **मूल** | सामान्य: "तुम्ही एक उपयुक्त सहाय्यक आहात" |
| **विशेषीकृत** | सविस्तर: "तुम्ही Zava DIY तज्ञ आहात, जे विशिष्ट उत्पादने शिफारस करतात आणि टप्प्याटप्प्याने मार्गदर्शन देतात" |

चालवल्यानंतर, तुम्हाला असे स्कोअरकार्ड दिसेल:

```
╔══════════════════════════════════════════════════════════════╗
║                    EVALUATION SCORECARD                      ║
╠══════════════════════════════════════════════════════════════╣
║ Prompt Variant    │ Rule Score │ LLM Score │ Combined       ║
╠═══════════════════╪════════════╪═══════════╪════════════════╣
║ Baseline          │ 0.62       │ 3.2 / 5   │ 0.62           ║
║ Specialised       │ 0.81       │ 4.1 / 5   │ 0.81           ║
╚══════════════════════════════════════════════════════════════╝
```

**प्रयोग:**
1. मूल्यांकन चालवा आणि प्रत्येक वेरिएंटचे स्कोअर नोंदवा
2. विशेषीकृत प्रॉम्प्ट अजून अधिक विशिष्ट करा. स्कोअर सुधारतो का?
3. तिसरा प्रॉम्प्ट वेरिएंट जोडा आणि ते सर्व तीन तुलना करा.
4. मॉडेल उपनाम (उदा. `phi-4-mini` विरुद्ध `phi-3.5-mini`) बदलून तुलना करा.

### प्रयोग 5 - आपल्या स्वतःच्या एजंटसाठी मूल्यांकन लागू करा

मूल्यांकन फ्रेमवर्क वापरून आपल्या एजंटसाठी टेम्पलेट तयार करा:

1. **आपले golden dataset ठरवा**: अपेक्षित कीवर्ड्ससह 5 ते 10 चाचणी प्रकरणे लिहा.
2. **आपला सिस्टम प्रॉम्प्ट लिहा**: एजंटसाठी इच्छित सूचना.
3. **मूल्यमापन चालवा**: मूळ स्कोअर मिळवा.
4. **पुनरावृत्ती करा**: प्रॉम्प्ट बदला, पुन्हा चालवा, आणि तुलना करा.
5. **मर्यादा ठरवा**: उदा. "0.75 एकत्रित स्कोअरपेक्षा खाली न पाठवा".

### प्रयोग 6 - Zava संपादक पद्धतीशी कनेक्शन

Zava Creative Writer च्या संपादक एजंटकडे पुन्हा पाहा (`zava-creative-writer-local/src/api/agents/editor/editor.py`):

```python
# एडिटर हा उत्पादनात LLM-as-judge आहे:
{"decision": "accept/revise", "editorFeedback": "...", "researchFeedback": "..."}
```

हा भाग 8 च्या LLM-जज सारखाच **संकल्पना** आहे, पण ऑफलाइन चाचणी संच्याऐवजी उत्पादन पाइपलाइनमध्ये अंतर्भूत आहे. दोन्ही पद्धती:

- मॉडेलकडून संरचित JSON आउटपुट वापरतात.
- सिस्टम प्रॉम्प्टमध्ये परिभाषित गुणवत्ता निकष लागू करतात.
- पास/फेल निर्णय देतात.

**फरक:** संपादक उत्पादनात (प्रत्येक विनंतीवर) चालतो. मूल्यांकन फ्रेमवर्क विकासात (शिप करण्याआधी) चालतो.

## मुख्य मुद्दे

| संकल्पना | मुद्दा |
|---------|----------|
| **Golden datasets** | लवकर चाचणी प्रकरणे तयार करा; AI साठी रिग्रेशन चाचण्या आहेत |
| **नियमाधारित तपासण्या** | वेगवान, निश्चित, आणि स्पष्ट अयशस्वी घटना पकडतात (लांबी, कीवर्ड्स, फॉरमॅट) |
| **LLM-जज** | त्याच स्थानिक मॉडेलचा वापर करून सूक्ष्म गुणवत्ता स्कोअरिंग |
| **प्रॉम्प्ट तुलना** | सर्वश्रेष्ठ निवडण्यासाठी एकाच चाचणी संचावर अनेक वेरिएंट चालवा |
| **ऑन-डिव्हाइस फायदाः** | सर्व मूल्यांकन स्थानिक पातळीवर; API खर्च नाही, दर मर्यादा नाही, डेटा आपल्याकडून बाहेर जात नाही |
| **शिपपूर्वी मूल्यांकन** | गुणवत्ता निकष सेट करा आणि मूल्यांकन स्कोअरवर प्रकाशन नियंत्रित करा |

## पुढील पावले

- **विस्तार करा:** आपल्या golden dataset मध्ये आणखी चाचणी प्रकरणे व श्रेण्या जोडा.
- **स्वयंचलित करा:** मूल्यांकन आपल्या CI/CD पाइपलाइनमध्ये समाकलित करा.
- **प्रगत न्यायाधीश:** लहान मॉडेलच्या आउटपुटची चाचणी करताना न्यायाधीश म्हणून मोठे मॉडेल वापरा.
- **वेळेनुसार ट्रॅक करा:** प्रॉम्प्ट आणि मॉडेल आवृत्त्यांवर मूल्यांकन निकाल जतन करा आणि तुलना करा.
